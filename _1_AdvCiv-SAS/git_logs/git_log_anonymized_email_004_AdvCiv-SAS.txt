conservative performance optimizations to the keyboard navigation feature
As chatgpt 5.2 explains below (check if accurate, claude opus 4.5 review the change ok and excellent according to it):

That said: the only meaningful O(n) cost you still have with arrows is in SAS_navigateItemList(), because it re-scans self.list on every Up/Down press to rebuild the selectable indices + find current position. On a 2015 CPU this is already fine (lists are usually a few hundred entries), but if you want a safe + reliable micro-optimization, this one is good:

Safe optimization: cache “selectable rows” during placeItems()
What it changes

	- While placeItems() is already looping to draw the table, also build:
		- self.SAS_selectableListIdx: list indices of displayed real items (skips headers/spacers)
		- self.SAS_itemToSelectablePos: maps iItem → position in SAS_selectableListIdx
	- Then SAS_navigateItemList() becomes O(1) per arrow press (no list scanning), while preserving behavior with filtering/headers because the cache is built after filtering rules are applied.

Is this worth it?

	- Correctness risk: low (it’s just caching what you already displayed).
	- Speed gain: small but real (Up/Down becomes constant-time).
	- Compatibility: Python 2.4-safe (no ternaries/comprehensions; I used has_key in the fallback to match old Py2 style, but you can keep idx not in dict if you prefer).
