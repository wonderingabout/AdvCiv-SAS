# Main Changes Guide

This guide highlights key differences between AdvCiv-SAS and AdvCiv 1.12. It’s intentionally practical and a bit verbose so newer players can follow the "what changed" and "why it matters." For technical details, see the main [README](/README.md) and the docs under [/_1_AdvCiv-SAS/](/_1_AdvCiv-SAS/).

Many of these changes are partially or entirely tunable via [`GlobalDefines_advciv_sas.xml`](/Assets/XML/GlobalDefines_advciv_sas.xml).

- **Player-tunable options (SAS defines):** When this guide mentions DLL code locations (for example `CvUnitInfo::getTerrainAttackModifier`), that is just to document where the behavior comes from. **Players do not need to modify or recompile the DLL** to change the behavior; you can instead change/tune the corresponding **SAS defines** (in the above mentioned **XML** file): for example the `SAS_CV_UNIT_INFO_ENABLE_UNIT_TERRAIN_ATTACK_MODIFIERS` SAS define can be set to `0` to disable or `1` to enable this option. Note: after modifying them, restart Civ4 to apply changes.
- **Scope note:** The long-term goal is to make more gameplay options player-tunable via SAS defines when it is practical. Some changes are still hardcoded (or not worth exposing) due to complexity, risk, or limited value, so not everything is configurable.

Note: The main changes guide serves as an index that covers concisely most AdvCiv-SAS changes since as of now base AdvCiv 1.12, but it may not be exhaustive, see the other AdvCiv-SAS supporting docs or files for further information.

## Main Changes

### Not supported in AdvCiv-SAS

Some features are not supported in AdvCiv-SAS; e.g., non-English languages, pre-AdvCiv-SAS scenarios. See [root README.md: Not supported in AdvCiv-SAS](/README.md#not-supported-in-advciv-sas). (Requires AdvCiv-SAS 5971+) See also [KI#166](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#166---fixedaddressed-base-advciv-issue-of-trying-to-support-reading-mod-eg-advcivadvciv-sas-replayhall-of-fame-data-in-unmodded-bts-or-unmodded-bts-replayhall-of-fame-data-in-the-mod-eg-advcivadvciv-sas-replays-no-longer-try-to-use-vanilla-bts-compatible-replay-storage-after-shifted-xml-enum-order).

### Renaming

- "Play Now" → "Simple Game" Clearer, matches code name, better contrasts with "Custom Game", and same as in C2C mod.
- "WFYABTA" → "We fear you are trading more than us." Same mechanic; the new text reflects that it’s about trading volume, not tech pace.
- Fixed misleading event text such as "The forge has been destroyed" used even when building actually survives (e.g. "The forge has caught fire" instead).
- Foreign Advisor's "Active" tab -> "Treaties" tab (no longer shows redundant bonuses).
- "All Units" → "No UnitCombat" in the Military Advisor: was inaccurate: this does not list all units (e.g. military units such as swordsmen), but only `<Combat>NONE</Combat>` units (e.g. Worker, Workboat, Missionary).
- Rename UnitCombats for concision or accuracy: "Recon Units" -> "Recon"; "Siege Weapons" -> "Siege"; "Helicopter Units" -> "Helicopters"; "Armored Units" -> "Armor"; "Naval Units" -> "Naval"; "Gunpowder Units" -> "Gun" (was inaccurate since gunpowder is no longer used in modern guns).
- Nuclear interception text clarifies that displayed chance is **adjusted** by nuke evasion; see the forum thread for context ([civfanatics](https://forums.civfanatics.com/threads/sdi-icbm-and-tactical-nukes.239415/)).
- "Civilopedia" → "Sevopedia".
- "Ressource(s)" → "Bonus(es)".
- "Difficulty" → "Handicap" (matches Civ4 naming).
- Renamed "Horseback Riding" to "Mounted Combat" (includes all mounts, not just horse-based ones).
- Bonus names standardized: Ivory → Elephants; Cow → Cattle; Clam → Molluscs; Wine → Grapes, etc.
- Terrain/Feature: Ice (Terrain) → Ice Sheet; Ice (Feature) → Ice Cap.
- "Great wonders" → "World wonders" (functionally the same).
- Leader names corrected where needed (e.g., Montezuma → Moctezuma II).
- Taoism → Daoism (with matching building/unit names and encyclopedia text).
- Cho-Ko-Nu → Zhuge Nu (non-standard/incorrect Wade–Giles romanization; now using modern pinyin).
- "Unique units" → "Civilization units" (they aren’t singletons; they’re civ-specific).
- Cavalry → Dragoon (too generic/era-agnostic; new name and unit better matches early-industrial role).
- Promotion names made more explicit (e.g., Counter-Archer, Counter-Siege, City Bombard Damage). Roman numerals → Arabic ("Combat 3").
- "Demographics" → "Info Screen" (correct advisor name: does not only show the Demographics tab of the Info Screen) (Sevopedia Shortcuts)
- "Victory Status" and "Victory Conditions" → "Victory Screen" (similar inconsistent naming in Sevopedia Shortcuts)
- "Units" -> "Start Scroll at Units (Production Chooser Bar)" (City Screen) (clearer and so it is not confused with the new AdvCiv-SAS Building Filter buttons' behaviour (filter on click vs set default scroll position))

### Code/Performance optimizations

- **Launch guard for SAS defines (startup/load/Sevopedia):** Added a strict define-integrity check using one **int sentinel** and one **string sentinel**, validated at pre-game start, game start, save load, Sevopedia creation, and Sevopedia show. If a value is missing or mismatched, Python raises a **fatal error** with `actual` vs `expected` values and guidance to launch from a Windows shortcut. The game will be **unplayable if the config is incorrect**. Change in: [`SASDefineGuard.py`](/Assets/Python/SASDefineGuard.py). See also [Quick Install Guide](/_1_AdvCiv-SAS/Docs/README_Quick_Install_Setup_Guide.md#place-a-shortcut-of-the-mod-on-your-desktop).
- **Lazy init and cache SAS defines** (e.g., in advisors) to `None` and then to their actual value once if cache is `None`. This prevents and **empirically fixes**  direct caching in init, that empirically causes **python-like crashy behaviour** (i.e. similar behaviour to when python code is changed during runtime) on XML SAS define change of a value (instead of no effect and requiring a restart for the change to take effect). See for example [CvMilitaryAdvisor.py](/Assets/Python/Screens/CvMilitaryAdvisor.py)'s `initDefines` method. Possibly similar issues and crashes encountered when early caching sevopedia playground's background and cache. See [KI#128](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#128---seemingly-fixed--worked-around-runtime-ui-definestyle-changes-could-produce-crashy-python-like-behavior).
- **Fail-fast XML lookup hardening:** replaced almost all `getInfoTypeForString(...)` calls (terrain/features/bonuses and similar tags) with `getInfoTypeOrFail(...)` (in [SASUtils.py](/Assets/Python/SASUtils.py)), so **missing/renamed XML tags raise explicit errors** instead of silently using invalid `-1` values. Notably allowed to detect and fix incorrect `getInfoTypeOrFail("DIRECTION_NORTH")` instead of correct `DirectionTypes.DIRECTION_NORTH` (in Planet Generator map), or incorrect `getInfoTypeOrFail("PLOT_PEAK")` failing instead of correct `PlotTypes.PLOT_PEAK` (in Peirce map). (Requires AdvCiv-SAS 5940+) Also added `findInfoTypeNumOrFail(...)` for older `CvUtil.findInfoTypeNum(...)` callsites that need getter/count function. Note: now also double-checked by a GitHub workflow check which now blocks new raw static/literal Python info-type lookups unless they are explicitly listed as dynamic/guarded legacy exceptions.
- **Static define-key audit:** added [`audit_define_keys.py`](/LLM_Helpers/audit_define_keys.py), which statically greps every literal `getDefine{BOOL,FLOAT,INT,STRING}("X")` call across Python and DLL and verifies `X` is declared in mod XML, base BTS XML, or vanilla Civ4 XML. Catches **per-key drift** (renamed/removed XML entries) before runtime. A runtime fail-loud wrapper isn't practical here because `getDefine*` returns `0`/`""` for both missing and legitimately zero/empty keys (unlike `getInfoTypeForString` which has a clean `-1` sentinel); a static grep is also **cheaper and simpler to maintain** than wrapping every call site. See [LLM_Helpers (README.md)](/LLM_Helpers/README.md).
- **Unused-text-key audit + cleanup:** added [`audit_unused_text_keys.py`](/LLM_Helpers/audit_unused_text_keys.py), which flags mod GameText `<Tag>TXT_KEY_*` entries referenced **nowhere** in the mod, base BTS, or vanilla Civ4 (Python + DLL + XML); dynamically-built (`"TXT_KEY_BUG_OPT_"+id`) and engine-derived (`_PEDIA/_HELP/...`) key families are routed to a separate review bucket to avoid false positives. Used it to **remove confirmed dead GameText entries** from [AdvCiv-SAS_main.xml](/Assets/XML/Text/AdvCiv-SAS_main.xml). See [LLM_Helpers (README.md)](/LLM_Helpers/README.md).
- (Requires AdvCiv-SAS 5961+) **C++ signature cleanup:** collapsed **safe multiline C++ function declarations/definitions** to one physical line across the DLL source, making **signatures easier to grep, compare, and review** in diffs/LLM-assisted passes. Done with the help of the new [`collapse_cpp_signatures.py`](/LLM_Helpers/README.md#collapse_cpp_signaturespy), a **conservative** signature-only helper, which intentionally skips ordinary multiline calls and unsafe comments, while also supporting traceable hoisting for reviewed whole-line comments inside signatures; review its output as a mechanical source-cleanup tool rather than a general C++ formatter. It was empirically idempotent after the first full DLL pass (rerunning reported no safe C++ signature collapses found). This is especially useful for **overloaded functions** like `void addMessage`, or e.g., `void CvDeal::endTrade(TradeData trade, PlayerTypes eFromPlayer, PlayerTypes eToPlayer, bool bTeam, /* advc.036: */ bool bUpdateAttitude, PlayerTypes eCancelPlayer) // advc.130p`, instead of its old three-line equivalent.
- (Requires AdvCiv-SAS 5961+) **C++ inline-return cleanup:** collapsed simple inline DLL header functions whose bodies were exactly one `return` statement to one physical line, making trivial getters/wrappers easier to grep and compare. Done with the help of [`collapse_cpp_inline_returns.py`](/LLM_Helpers/README.md#collapse_cpp_inline_returnspy), a conservative mechanical helper that skips constructors/destructors, multi-statement bodies, body comments, multiline signatures, and overlong results. For example, `int getGoldPerTurnByPlayer(PlayerTypes ePlayer) const { return m_aiGoldPerTurnByPlayer.get(ePlayer); } // Exposed to Python`, instead of its old four-line equivalent.
- (Requires AdvCiv-SAS 5972+) **Build cleanup - disabled unneeded DLL test code in normal builds:** compile-gated disabled reproducibility-test code (`ReproTest.cpp`) so normal DLL builds no longer compile it unless explicitly enabled for debugging. This keeps dormant test/debug code out of normal builds, may slightly reduce DLL/build size, and does **not** change gameplay. Found with the help of the new [`find_cpp_dead_code_candidates.py`](/LLM_Helpers/find_cpp_dead_code_candidates.py) LLM/manual-review helper.
- (Requires AdvCiv-SAS 5930+) **GitHub workflow build checks:** added GitHub Actions checks under [`.github/workflows`](/.github/workflows/) to catch forgotten debug/default settings and data drift at each commit. Current checks notably verify launch-guard sentinel values, XML tag references in SAS define text values, integer SAS define bounds, valid main-menu opening-music script setup, disabled BBAI logging defaults, shared UI font defaults, Sevopedia Leader AI Personality Panel predumped-cache defaults and predump file presence, `WorldSizeTypes` enum/XML alignment, `CvGameCoreDLL/Project/temp_files/` staying effectively empty (only its tracked placeholder file), not being ignored by `.gitignore`, and being allowed only as an archive-excluded development marker through `.gitattributes` (helps catch stale fast-compile temp files. We check this because we suspect fast compiles can be unreliable, and also helps catch forgetting to replace the committed DLL after compiling; see [KI#38](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#38---no-idea-why-but-sometimes-compile-mysteriouslystrangelyinconsistently-fails-and-recompiling-succeeds-underwith-but-exact-same-source-files-if-i-am-not-mistaken-but) and [KI#38.2](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#382---weird-dll-xml-errors-at-launch-solved-by-recompiling-the-exact-same-dll-cosmic-ray-2-or-something-else-maybe-or-whatever-maybe-but)), and every playable `PrivateMaps/*.py` being listed exactly once across the SAS map-script heaviness/coverage defines while every listed map script exists in `PrivateMaps/`. For example, this helped spot [map scripts that were previously unclassified in SAS map-script heaviness defines](https://github.com/wonderingabout/AdvCiv-SAS/actions/runs/27198308080/job/80295526028); they are now listed explicitly for exhaustiveness (more robust; no gameplay change).
- (Requires AdvCiv-SAS 5975+) **AIP predump refresh bot:** added [`aip-predump-refresh.yml`](/.github/workflows/aip-predump-refresh.yml), a tested GitHub Actions bot workflow for the Sevopedia Leader AI Personality predump cache. The existing checker still fails when the committed predump is stale, but the refresh workflow can now run `aip_predump_values.py --write` outside Civ4 and open/update a bot PR containing only `SevoPediaLeaderCachePredumped.py` when real generated AIP data or display-label drift is detected. It runs automatically after pushes to known AIP predump dependency files. Tested in [PR #31](https://github.com/wonderingabout/AdvCiv-SAS/pull/31) with no-op, XML comment-only, numeric Leader XML, and Python label/display drift cases, so harmless comments do not refresh the cache while real generated predump changes do.
- Other examples: (Requires AdvCiv-SAS 5935+) This notably also **helped spot and fix duplicate text, parent, and child XML keys** found by new GitHub workflow check [text](https://github.com/wonderingabout/AdvCiv-SAS/actions/runs/27332010214/job/80746523894), [parent](https://github.com/wonderingabout/AdvCiv-SAS/actions/runs/27258698041/job/80498936912), and [child](https://github.com/wonderingabout/AdvCiv-SAS/actions/runs/27336768909/job/80762873956) (See [KI#148](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#148---fixed-base-advciv-issue-and-one-advciv-sas-issue-of-duplicate-parent-xml-keys-found-by-new-github-workflow-check), [KI#149](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#149---fixed-base-advciv-issue-duplicate-xml-text-tags-found-by-new-github-workflow-check), [KI#150](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#150---fixed-base-advciv-issue-and-some-advciv-sas-priority-duplicate-xml-childlist-entries-found-by-new-github-workflow-check)), and [suspicious malformed-looking XML tag punctuation](https://github.com/wonderingabout/AdvCiv-SAS/actions/runs/27342682844/job/80783239087) (e.g., `<French>>`, `</English>>`) (see [KI#151](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#151---fixed-base-advciv-issue-suspicious-malformed-looking-xml-angle-tags-found-by-new-github-workflow-check)) and [XML suspicious text chars](https://github.com/wonderingabout/AdvCiv-SAS/actions/runs/27400470104/job/80977135402) (see [KI#152](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#152---fixed-suspicious-replacement-question-marks-in-lengthy-sevopedia-xml-found-by-new-github-workflow-check)). ## AI

### AI — General behaviour

#### General changes (AI)

- Global AI settings lean toward **pragmatic, opportunistic** play over role-play: religion weight adjustments, lower revolt chance, lower war anger, more willingness to demand tribute (and sometimes declare if refused), and less inclination to take bad wars (especially multi-front). For exact values, see [`AI_Variables_GlobalDefines.xml`](/Assets/XML/AI_Variables_GlobalDefines.xml), [`GlobalDefines_advc.xml`](/Assets/XML/GlobalDefines_advc.xml), and [`BBAI_Game_Options_GlobalDefines.xml`](/Assets/XML/BBAI_Game_Options_GlobalDefines.xml).

#### Victories (AI)

- (Requires AdvCiv-SAS 5982+) **Cultural-victory AI is viability- and bottleneck-focused:** The new `SAS_BBAI_CULTURE_LOG_LEVEL` BBAI logging showed Base AdvCiv could treat an AI raising the culture slider only for happiness or border pressure as evidence that it was pursuing a cultural victory. Even the early preparation stage could then raise culture priorities in every city. As a result, the AI could spend production, specialists, and commerce on cities culturally ranked lower than the cultural-victory target group (i.e., as of now the top 3 cities that must reach Legendary culture), on a target city already much closer to Legendary than another target city, or even on a city that had already become Legendary. This was inefficient and could leave the AI dangerously underinvested in stronger economic, scientific, or military priorities. To address these issues, the AI no longer treats its culture slider alone as a cultural-victory commitment, and early preparation no longer raises culture priority throughout the empire. Before serious cultural victory pursuit begins, at least 2 of the 3 target cities must now each have already accumulated a tunable share of the Legendary requirement (10% by default), and the AI must rank among the configured number of leading cultural contenders (No point pursuing culture if we won't win culturally anyway). Once committed, only the target cities receive victory-specific culture weight, shifted toward the city projected to take longest (no point having a city have 70% legendary culture and another 10% legendary culture only, as several legendary cities are required to win culturally, so have the spread be more even); a city that is already Legendary keeps only independent local-border value. Converting production to culture remains available for first-ring expansion and strongly contested borders, but otherwise only to active target cities.
- (Requires AdvCiv-SAS 5982+) **Hopeless late cultural pursuits are abandoned:** The new `SAS_BBAI_CULTURE_LOG_LEVEL` BBAI logging also showed a late cultural pursuit in which the least advanced target city was projected hundreds of turns from Legendary when the estimated game had little more than 100 turns remaining (e.g., Future Era at T~400 at Normal game speed), despite likely faster Space or other victory routes. The AI now returns to preparation when its maximum affordable culture-slider projection misses the estimated deadline beyond the tunable grace or falls too far behind competing cultural contenders, freeing resources for stronger victory routes.
- (Requires AdvCiv-SAS 5983+) **Cultural pursuit gated by strong current or planned war enemies:** no point investing heavily in a future cultural victory if the AI may lose the war first. The AI no longer pursues serious culture-victory if its current or planned war enemies reach the shared `bEnemyStrong` power threshold (>=120% enemy power ratio by default) (the local `bDanger` is intentionally absent, because one threatened city should not veto an empire-wide victory strategy), reducing culture-victory pressure on production, specialists, and commerce while preserving ordinary local-border culture. This is not an absolute ban but relative to military high enemy power and cultural advancement: Culture 2 can still remain available against weaker current or planned war opponents, culture can resume after peace if relevant, and a close late cultural win is not abandoned if all target cities already have credible high-culture countdowns. In BBAI tests, a midgame map test showed the rule firing 25 times for Spain at 124% to 288% enemy power, blocking Culture 2+ pursuit despite otherwise meeting the new culture-victory foundation gates; Culture 2 remained available during a weaker war at 67% enemy power and resumed for four consecutive turns after peace.
- (Requires AdvCiv-SAS 5984+) **AI prioritizes a more advanced victory plan over an unprepared cultural bid:** when an AI has advanced far enough (a tunable AI Victory Stage as of now >= 3 (out of max 4) by default) toward another victory (Space, Conquest, Domination, or Diplomatic), it no longer diverts production, specialists, and commerce into serious cultural-victory spending unless all 3 required Legendary-city candidates are already advanced enough to keep culture credible. Ordinary local-border culture remains unaffected, and an AI already close to winning culturally can still finish the job. In a confirming map-437 BBAI test, this stopped Rome's serious cultural spending 17 times after it reached Domination 3, when only 1 of its 3 required culture cities was advanced and a cultural win remained 499 to 737 turns away. Rome could still experiment with culture before Domination 3, while Spain remained free to pursue culture because it had no similarly advanced alternative victory.

#### Bonuses (AI)

- **Strategic bonus "AI Objective" weighting (rework):** Now assign non-zero `iAIObjective` to key bonuses (Copper, Iron, Horse, Camel, Aluminum, Uranium). In base AdvCiv these were `0`, so logic that referenced them did nothing; now they actually tip decisions. We also added code paths that consume these weights: (1) **city-site scoring** (`AIFoundValue::evaluate`) — extra-value candidate sites that have these bonuses in their BFC; (2) **worker build priority** (`CvUnitAI::AI_bestCityBuild`) — e.g., improve/connect Copper before Stone; (3) **which city to improve next** (`CvUnitAI::AI_nextCityToImprove`) — prefer cities with unconnected high-objective bonuses; (4) **trading iAIObjective bonuses** (see [Diplomacy (AI)](/_1_AdvCiv-SAS/Docs/README_Main_Changes_Guide.md#diplomacy-ai)). Display added in Sevopedia Bonus pages; see there or XML for details.

#### Civics (AI)

- **Civic "Percent Anger" logic (rework):** Fixed the AI tendency to adopt any civic with `iCivicPercentAnger` (e.g. the former CIVIC_EMANCIPATION) just because rivals have it, regardless of its actual value. Replaced with an **opportunistic per-city check**: only consider such civics when enough of our cities would directly **lose anger** from the switch. Implemented in `CvPlayerAI::AI_civicValue`. In autoplay before/after tests, civic choices appeared **stronger**. See [KI#75](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#75---tremendously-improved-ais-autopicking-civic_emancipation-icivicpercentanger-just-because-other-rivals-have-it-regardless-of-how-good-the-civic-itself-is-now-replaced-with-opportunistic-current-unhappiness-per-city-based-logic-if-it-benefits-us-in-cvplayeraiai_civicvalue).
- **Civic choice logic — unlimited specialists generalized:** Addressed the AI’s tendency to default to **Caste System** and stay there. Rewrote `CvPlayerAI::AI_civicValue` so **unlimited specialists** are scored by **yields** (Food > Hammers > Gold) and **commerce** (Beakers = Gold > Espionage > Culture), instead of effectively crediting only culture. This lets civics like **Wage Labor** (unlimited Engineers) be chosen when appropriate. Added "high-need" boosts (e.g., extra Culture weight when pushing culture/BFC (quite similar to old logic); extra Engineer/Hammer weight when enemies are strong) and **anarchy-cost awareness** (devalue switches that cost many turns). Leads to a more even spread of civics among AIs based on their strategic needs, and to them being more responsive to change. Details/screenshots: [KI#76](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#76---tremendously-improved-ais-almost-always-picking-civic_caste_system-and-then-almost-never-changing-it-no-matter-what-unlimited-specialists-logic-favoured-only-culture-as-well-and-much-needed-an-improvement-and-generalization-in-cvplayeraiai_civicvalue). *(Observed in autoplay: early new Slavery<->Caste swaps; late much more Wage Labor and new occasional late Serfdom. May still subject to tuning.)*
- **Civic-switch damping (anti-oscillation):** Reworked `CvPlayerAI::AI_doCivics` to add a **percent-slack** gate: a new civic must outscore the current one by as of now **~10%** to trigger a switch. Goal: reduce rapid flip-flopping exposed after the broader civic valuation work. (Requires AdvCiv-SAS 5953+) Additionally, BBAI diagnostics showed the expensive remaining problem was **paid-anarchy civic churn**, especially direct reversals soon after the normal civic timer expired. Civic switches that add anarchy now need a larger value gain, and paid-anarchy revolutions add a longer cooldown before civics are reconsidered; free/no-extra-anarchy bundled changes remain more responsive. In the tested run, paid-anarchy revolutions dropped from 52 to 23, total anarchy turns from 174 to 81, and both-sides-paid direct reversals from 49 to 17 (See [KI#159](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#159---improved-ai-civic-switch-damping-paid-anarchy-civic-churn-and-direct-reversals-after-the-civic-timer-expired)). This should be stronger, considering excessive anarchy downtime is inefficient, especially with our as of now AdvCiv-SAS anarchy duration of 2 turns. *Note:* If the slack/cooldown is too high, AIs switch **too rarely** and **react more slowly**; if too low, oscillation returns and players with **anarchy-on-switch** waste turns. *Note 2:* in some cases bypassed/ignored by AIs entirely for some reason.
- **Civic valuation — specialist commerce accounted for:** Civics that grant **extra commerce per specialist** (e.g., `CIVIC_REPRESENTATION` → beakers/gold/culture) were undervalued and thus rarely chosen. Updated `CvPlayerAI::AI_civicValue` to apply SAS **commerce weights** and stronger **derived multipliers**, so large specialist counts are priced properly. *Observed in autoplay:* more Representation picks in mid/late game, with continued Hereditary Rule/Universal Suffrage variation. Adjust to taste.

- **Master–vassal tech overlap penalty:** In `CvPlayerAI::AI_techValue`, reduce the value of researching techs already known by our vassal(s) or master to discourage redundant research and encourage complementary paths/trades instead. See also [Diplomacy (AI)](/_1_AdvCiv-SAS/Docs/README_Main_Changes_Guide.md#diplomacy-ai). **Update:** also apply the same devaluation between **sibling vassals** in the self-research logic (if any of our fellow vassal(s) already has the tech, we value researching it less ourselves).
- **Military tech priority when weaker:** In `CvPlayerAI::AI_techValue`, give much more weight to key **military techs** when we’re weaker in power; economic or "peaceful" techs are of limited use if we’re likely to be conquered (and can end up effectively researched *for* our future conqueror). In one autoplay sample this seemingly in Ewuare not being able to sustain his early lead anymore. #### Religions (AI)

- **Religion valuation power-weighted:** Reworked `CvPlayerAI::AI_religionValue` so religions are mostly valued by how popular they are among militarily powerful civs, and no longer mostly based on the religion of civs the AI already likes. This makes it much less likely to switch into a fringe religion and end up hated by the *strong* civs, meaning fewer "random" self-sabotaging switches and better diplomacy with them (more trade, fewer wars, more stable blocs). The same power-weighted value is also used wherever religion is evaluated (state-religion civics, espionage "change religion" mission, and trade/demand "adopt religion X"). See [KI#82](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#82---tremendously-improved-ai-not-adopting-the-popular-religion-among-strong-rivals-thus-being-hated-and-more-likely-to-lose).

#### Diplomacy (AI)

Note: Most, if not all, diplomacy AI changes apply both to **AI–AI** and **AI–human** interactions (e.g. human master with AI vassal, or vice versa).

Note 2: Master–vassal(s) changes are intended to strengthen overall AI play by improving cooperation between masters and vassals, rather than simply over-boosting one party. The aim is that, in many cases, they also benefit vassals (or at least do not clearly reduce their own chances of winning) while being in the master's interest, since a stronger vassal is more likely to be useful.

- **Multi-war threat management — "emergency peace" pre-checks**: In `UWAI::Team::considerPeace` (`UWAIAgent.cpp`), add sanity checks to **seek peace** when either (a) the team is fighting **too many simultaneous wars** or (b) the enemies’ **combined power ratio** vs. us is high enough. Goal: **survivability and focus** (fewer dogpiles, more targeted campaigns) while keeping thresholds lax so plenty of wars still occur. In a sample run, Hatshepsut, strongest at ~turn 150, kept the lead to ~turn 200 with these checks (previously died before 200). See **[KI#65](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#65---seemingly-tremendously-improved-hatshepsut-ai-who-was-strongest-player-at-turn-150-is-fighting-many-wars-and-dies-before-turn-200-added-emergency-peace-sanity-pre-checks-in-uwaiteamconsiderpeace-in-uwaiagentcpp)**.
- **Tribute/counter-propose consistency fix for pure Vassal/Surrender offers:** base AdvCiv/BtS diplomacy would **reject** direct **Tribute/Demand** ("Not bloody likely"), yet clicking **"What do you want in exchange?"** would **accept** for the same demand (with nothing extra). So players had to do an unnecessary extra click to get the same vassal result. Fixed by making `CvPlayerAI::AI_considerOffer` accept this exact pure one-item vassal/surrender demand directly, and by making [CvDiplomacy.py](/Assets/Python/CvDiplomacy.py) not remember this narrow accepted case as a tribute demand memory (because this is rather a UI consistency fix). See [KI#108](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#108---fixed-base-advciv-diplomacy-inconsistency-ai-can-refuse-tribute-for-pure-vassalsurrender-then-accept-the-same-deal-through-what-do-you-want-in-exchange-with-nothing-added).
- **Fair bonus trade pricing (building-effect aware).** In `CvPlayerAI::AI_bonusTradeVal`, add a quick pre-check that scales a bonus’s trade value by how many **citywide building effects it actually unlocks with current techs** (granary/market/grocer/harbor/temple/monastery, etc.). Wonders/cathedrals ignored; less pronounced for coastal-building effects, as we assume to simplify many cities aren’t coastal and thus wouldn’t benefit. **Practical result (examples):** Maize/Wheat/Grapes are priced noticeably higher than Pigs/Cows; Molluscs beat low-impact foods; Iron is valued slightly less than before when it has no building bonuses (should still be valuable for units). **Effect:** shuts down early GPT farming, makes AI<->AI swaps fairer, and strengthens AI openings/midgame. See **[KI#66](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#66---fixed-and-greatly-enhanced-ai-overvaluing-bonuses-that-give-minor-effects-such-as-pig-or-cattlecow-1-health-only-etc-that-ai-would-value-the-same-as-maize-or-wheat-2-health-effectively-since-almost-all-cities-have-a-granary-and-very-early-or-grapes-from-grocer-or-molluscs-from-harbor-for-example-by-adding-a-pre-check-in-cvplayeraiai_bonustradeval-to-value-relatively-more-bonuses-the-more-effects-they-provide-from-buildings-if-we-have-their-required-tech-of-these-buildings-so-that-humans-cant-abuseexploit-it-with-aggressive-trading-to-become-super-rich-xd-if-i-may-say-but-and-also-in-ai-vs-ai-trading-each-ai-gets-best-or-more-value-out-of-the-effective-value-of-their-bonuses-at-least-more-so-now-if-i-may-say-but)**.
- **Bonus trade valuation uses `iAIObjective`:** In `CvPlayerAI::AI_bonusTradeVal`, the AI now factors each bonus’s `iAIObjective` weight directly into price—higher-priority strategic bonuses (per your weights) command more value in trades.
- **Era- & inventory-aware relative pricing for key strategic bonuses (Horse<->Camel, Camel<->Horse, ±Elephants):** Also in `CvPlayerAI::AI_bonusTradeVal`, trade values now adjust by what the buyer already has and by era. Examples: Medieval buyer with **Horse/Camel** but no **Iron** values Iron ≈ x2 (to unlock Knights); by Renaissance that premium fades (Muskets don’t need any bonus; Cannons only need Iron). **Elephants** are valued in Ancient/Classical, but discounted in Medieval (≈ x0.5, for buildings at most). **Copper/Iron** are discounted in Classical if the buyer already has Camel/Horse/Elephants; Horse and Camel are treated as partial substitutes early on. **Update:** this adjustment no longer applies to trades *within* the same master–vassal(s) locus, since it is generally in their shared interest to strengthen each other.
- **Do not buy dominated or equivalent strategic bonuses (era and bonus-aware):** strategic resource buying now uses explicit exclusion rules, so AI skips dominated/equivalent offers instead of considering them as valid buys. Examples: in Classical, Copper is not needed if we have Iron; in Classical metals and mounted lines can substitute each other so redundant buys are skipped; Horse and Camel are treated as equivalent in Classical/Medieval/Renaissance; in Industrial (before Modern), Horse has the edge (no Camel Dragoon), so Camel is skipped if Horse is already available, etc. Change in `CvPlayerAI::AI_bonusTrade`. See [KI#109](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#109---tremendously-improved-ai-bonus-trading-ai-very-inefficiently-buying-dominated-or-equivalent-strategic-bonuses-era-and-bonus-aware-exclusions).
- **Aggressive tech trade discounts for widely known techs:** AIs now sell technologies known by ≥1 of their rivals at **heavy discounts** (e.g. 25% of the original cost/value), recognizing that it is better to secure **some gold** immediately via a cheap sale than to hold out and risk getting **0 gold** if a rival sells it first; implemented in `CvPlayerAI::AI_dealVal`. To compensate for this higher trading efficiency, **AI research costs** (`iAIResearchPercent`) have been adjusted (reduced/increased at higher/lower handicaps). See [KI#83](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#83---tremendously-improved-ais-not-trading-techs-more-aggressively-and-cheaply-when-rivals-discover-them-too-and-there-is-a-risk-they-will-beat-us-to-the-sale-leaving-us-with-nothing-rather-than-little-gold-or-such-which-would-have-been-much-better).
- **Preferential master–vassal tech trading:** Heavily increase willingness to trade techs with our **master/vassal(s)**; change in `CvPlayerAI::AI_doDiplo`. This synergises with the earlier "devalue techs your master/vassal already knows" change (see [Technologies (AI)](/_1_AdvCiv-SAS/Docs/README_Main_Changes_Guide.md#technologies-ai)) and extends it **to *all* techs**, not just techs that overlap with what our master/vassal(s) already know. In one autoplay sample this *seemed* to **reduce tech overlap** and **improve sharing** between vassals (who had many more techs) and their master (who now picks up more of them). Also updated `AI_contactRoll(CONTACT_TRADE_TECH` to pass an explicit probability multiplier (rather than relying on the default of 1), making tech-trade frequency more dynamic and tunable instead of solely personality-driven. See [KI#78](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#78---improved-trade-techs-preferentially-with-our-vassals-or-master-synergises-with-the-no-overlap-previous-master-vassals-tweak--add-a-second-parameter-now-ai_contactrollcontact_trade_tech-rcontactprobmult-to-have-a-more-dynamic-tech-trading-not-only-based-on-personality-according-to-chatgpt-51-and-then-corrected-by-claude-sonnet-45s-review-check-if-accurate). **Update:** Extend the same extra trade bias to **sibling vassal<->vassal** trades.
- **Military-power–aware tech trade targeting:** If a potential tech-trade is *not* between members of the same master–vassal(s) locus, `CvPlayerAI::AI_doDiplo` instead increases tech-trade contact chance toward **militarily weaker** players (safer, less likely to threaten us) and reduces it for **stronger** ones (less free value to future rivals). In one autoplay sample this seemingly produced a much closer game: the former runaway (Ewuare) was repeatedly caught up by rivals, and the game lasted to ~T419 instead of ~T288, with the lead changing hands several times. See [KI#80](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#80---tremendously-improved-ai-contacting-for-tech-trades-players-that-are-stronger-even-though-this-is-more-likely-to-be-detrimental).
- **Master–vassal(s) locus first for tech trades:** Whenever the AI wants to contact other players for a tech trade in `CvPlayerAI::AI_doDiplo`, it now **first checks** whether the desired tech is already tradeable inside its own master–vassal locus (master, vassals, sibling vassals). If so, it **does not ask outside civilizations for that tech**, nudging trades to stay within the group instead. In one autoplay sample this *seemed* to help Ewuare win earlier by finally picking up the Dragoon-related military tech from a vassal, while the locus kept a similar overall tech pace to previous runs. See [KI#79](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#79---improved-before-contacting-other-players-for-tech-trades-first-check-if-we-dont-already-have-the-tech-in-our-master-vassals-locus-and-if-so-dont-contact-the-other-players).
- **Military priority in tech trades when behind:** When evaluating the value of a tech in trades with players outside the AI’s master–vassal(s) locus in `CvTeamAI::AI_techTradeVal`, the AI values **key military techs** (high `FLAVOR_MILITARY`) more highly if it is **weaker** than its rivals (they matter more for survival than additional economic techs); as of now most such military techs are **untradeable**, which is consistent with autoplay impact being small, but this may matter more if some of them become tradeable.
- **Devaluation of locus-known techs from outsiders:** When evaluating the value of a tech in trades with players *outside* the AI’s master–vassal(s) locus in `CvTeamAI::AI_techTradeVal`, the AI values techs that any member of its locus already knows **moderately less** when they are offered by outsiders (since that knowledge is already available within the group). In one autoplay sample this *seemed* to contribute to more decisive wins for AIs that had accumulated more vassals.

#### Buildings (AI)

- Tightened/reworked **wonder-keep** thresholds in `CvCityAI::AI_chooseProduction`: if at war/planning war/in danger and not close to completion, AIs will **ditch world wonders** (highest threshold when weaker) and redirect hammers to urgent needs (e.g., walls/units). See **[KI#48](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#48---enhancedreworked-ai-building-walls-when-they-are-stronger-and-dont-need-it-or-wonders-when-they-are-weaker-or-in-danger-dont-build-wonders-for-our-neighbours-when-they-capture-us-and-overall-added-a-lot-of-extra-buildingvalue-reject-or-always-build-first-logic-depending-on-building-type-or-such-and-reworked-as-well-the-ditching-wonders-logic-to-be-seemingly-stricter-and-more-wardanger-focused)**.
- Do not build **Anti-Barbarian Border Blocking** (`bBorderObstacle`) **Wonders** (e.g. Great Wall) past a certain **era** (e.g. Classical era, meaning we will not build Great Wall or equivalent wonders from the Medieval and onwards). Change in `CvCityAI::AI_buildingValue`.
- Added **pre-filtering** rules in `CvCityAI::AI_buildingValue` so hammers go where they matter: skip walls when safe/strong; avoid wonders when weak/in danger (focus on core cities if relevant); keep early culture to BFC needs; do **not** block economy/growth buildings; fall back to base logic after the pre-filters. AIs tend to value **Heroic/National Epics** more and build them in top-hammer cities. See **[KI#48](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#48---enhancedreworked-ai-building-walls-when-they-are-stronger-and-dont-need-it-or-wonders-when-they-are-weaker-or-in-danger-dont-build-wonders-for-our-neighbours-when-they-capture-us-and-overall-added-a-lot-of-extra-buildingvalue-reject-or-always-build-first-logic-depending-on-building-type-or-such-and-reworked-as-well-the-ditching-wonders-logic-to-be-seemingly-stricter-and-more-wardanger-focused)**. Does **not** affect Buildings with a **negative or equal to 0 XML `iCost` (Shrines, Corporations, etc.)**, since we don't need to save hammer for free of cost buildings, and they would otherwise not be built or heavily delayed if affected by any SAS `CvCityAI::AI_buildingValue` gate.
- Regardless of the aforementioned pre-filtering rules, build **cheap World Wonders ASAP** (hammer `iCost` gates tunable by era in SAS defines): early, we want to prioritize most hammer towards early urgent growth (settlers, workers, etc.) or military (urgent rush or defense); later on they **cheap cost** (e.g. Pyramids not yet built Medieval era) makes it attractive and worth having ~1-2 less units or similar. Change in `CvCityAI::AI_buildingValue`. Note: about Wonders generally, AdvCiv-SAS uses a fairly strict low wonder policy, as early settlers and workers are more valuable generally and a stronger choice; plus later on we have little time to build world wonders as we risk being overrun by our rivals and building world wonders for them as they conquer us, which is a poor choice. A mostly Base AdvCiv-like behaviour of AIs building many wonders can be restored in SAS defines by disabling `SAS_AI_BUILDING_VALUE_WORLD_WONDERS_OPTIMIZE` (e.g. 0 vs 6 wonders at turn 100 in an autoplay sample with vs without it), but this should not be necessary and not recommended to change SAS building wonder defaults to have an optimal AI play, as this build cheap wonders ASAP change causes most wonders to built by turn 200 (when it is more efficient to do so). Adjust as you see fit.

- **Unit upgrades - stop wasting gold on tiny gains:** Fix AI tendency to **over-upgrade units** even when the upgrade is **poor value** (high gold cost for a small strength gain), which can burn many turns of **gold** and **slow tech pace**. Before allowing an upgrade, the AI must pass **simple ROI gates** based on **strength** gained vs gold cost: (1) gold per 10% strength gain, and (2) gold per +1 **flat strength** gain; both gates are **more lenient with higher unit EXP** (high-EXP units keep their promotions after being upgraded which is valuable, so spending more gold on them is more justified), while low-EXP units are treated more strictly (often better to let them die and rebuild if needed, unless it is cheap enough to upgrade them). Example (Knight -> Cuirassier): 10 strength -> 13 strength; if one of the gates requires `15 gold per +1 flat strength gain` with `+1 gold per +1 flat strength gain per 1 EXP`, then we won't upgrade the unit if its upgrade cost is more than `15 * 3 = 45 gold` at **0 EXP**, or `((15 + 18) * 3) = 99 gold` at **18 EXP** for this gate. This is thought to be much stronger, because it uses the current military advantage before enemies catch up; old units can be spent in war instead of sitting around draining upkeep, needing future upgrade gold (solved if they die as well), and risking obsolescence; successful captures can then increase economy, unit support, and long-term growth; and the needless wait was very problematic and inefficient too. See [KI#156](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#156---fixed-base-advciv-issue-ready-city-attack-stacks-could-park-for-future-upgrades-even-when-no-unit-could-upgrade-now).

For example, in BBAI diagnostics, an AI with 280 gold reserved for upgrades could spend 237 gold, still count as under budget, then start another 237 gold upgrade and end up spending 474 gold total, sometimes leaving only a few gold in the treasury. Routine **upgrades now check the actual remaining upgrade budget before upgrading**. **Emergency**-style upgrades, such as important city defenders or special impassable-unit upgrades, **remain more flexible**, but normal upgrades are much less likely to waste all available gold. - (Requires AdvCiv-SAS 5952+) **AI attack stacks now opportunistically capture nearby barbarian cities:** before this fix, ready city-attack-capable (i.e. not just `UNITAI_ATTACK_CITY`) stacks could sit idle because the AI had **no normal war target yet**, even with a **nearby pathable barbarian city**. This does not override normal wars or normal war targets; it fills the no-target idle gap. This change is usually a **strong** move: barbarian cities are often weak, cost little diplomatically, and capturing them can give land, population, resources, unit support, and future growth without spending a settler or interrupting city production nor growth. In the tested save before the fix, nearby barbarian cities survived at game end at ~T400 despite reachable AI armies to Napoleon and Shaka, and barbarian being very weak in comparison, yet bonuses-rich, and Shaka was among weakest players; after the fix, Zulu captured Hurrian at T127, Alemanni at T132, Visigoth at T148, and Gaul at T153, turning those nearby barb cities into an early expansion boost and making Shaka the strongest player for almost all middle game and end game, almost allowing him to win. This change should make AI significantly stronger. ## General changes

- **Start map behavior**: less "terrain polishing"; more starting **vision** so you can choose a spot rather than forcing ideal tiles.
- **Slavery**: `HURRY_POPULATION` production increased (**24 → 27**). See [CIV4HurryInfo.xml](/Assets/XML/GameInfo/CIV4HurryInfo.xml) (`HURRY_POPULATION`).
- **Leader random selection**: `PER_EXTRA_LEADER_CIV_SELECTION_WEIGHT` **20 → 0** (all civs equal chance regardless of leader count). See [GlobalDefines_advc.xml](/Assets/XML/GlobalDefines_advc.xml).
- **We Love the King Day**: restored (present in AdvCiv-SAS).
- **Cost rounding to 5 removed (human-side)**: Eliminated the base AdvCiv "round to 5" rule from `CvPlayer::getProductionNeeded` (units, buildings, projects) that applied only to human players, so **costs are no longer altered for the human player** — e.g., a Swordsman at 42 hammers costs 42 (not 40), at 43 costs 43 (not 45), and an Ancient Maceman at 18 costs 18 (not 20). Also removed analogous rounding in `CvPlayer::greatPeopleThreshold`, `CvPlayer::getNewCityProductionValue`, and `CvTeam::getResearchCost`. This fixes the Sevopedia vs. in-game mismatch, and the issue of human players having a different cost than AI players even if cost would have otherwise been the same (e.g., at Noble, the AI paid 42 while the human paid 40; 43 vs. 45). Fine-grained costs now also enable true **per-hammer** tuning—e.g., you can set 40, 41, 42, 43, 44, or 45 instead of being forced to choose 40 or 45. See [KI#67](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#67---fixedenhanced-an-ancient-maceman-18-hammer-costs-20-hammer-ingame-and-a-swordsman-42-hammer-costs-40-while-a-swordsman-43-hammer-costs-45-which-is-a-mess-fixed-by-removing-per-5-rounding-in-cvplayergetproductionneeded-and-other-related-functionsissues).
- **Hints**: remove multiple copies of the same hint. Change in [CIV4Hints.xml](/Assets/XML/GameInfo/CIV4Hints.xml).
- **Mod scope:** keep overall gameplay relatively lightweight (no clutter), but go deeper on the systems that are changed.

## Game Speeds

- `GAMESPEED_MARATHON` **shortened** total turns **1250 → 1000** (vs base AdvCiv). Adjusted the associated speed modifiers to align with the new length.
- **New Game Speeds**: `Slow (1500 turns / 300%)` and `Very Slow (2000 turns / 400%)`, **slower than Marathon (1000 turns / 200%)**; and `Turbo (250 turns / 50%)` and `Nitro (165 turns / 33%)`, **faster than Quick (330 turns / 67%)**, that should provide varied experiences. See full game speed details in the **new Sevopedia Speeds category**.
- **Clarified** and made more **outwardly visible** the **Game Speed** information: e.g. `"Normal (500 / 100%)"` bullet; `Epic (750 / 150%)` bullet; `GAME SPEED (50 000 BC - 2105 AD)` header, and clarified other fields.
- **Reordered** them in XML from fastest to slowest so they appear in this order in the UI (Main Menu)'s **Simple Game** Game Speed section.
- **Fix base AdvCiv `iExtraFreeOutsideUnits` incorrect default bug:** Faster-than-Marathon speeds ended up with **100 extra free outside units** due to two issues: the `iExtraFreeOutsideUnits` XML tag was missing for those speeds, and the DLL fallback default for that tag was incorrectly set to **100** (Marathon explicitly had **1**). Fixed by changing the DLL default from 100 to **0**, and by adding the missing `iExtraFreeOutsideUnits` game-speed XML field with a value of **0** so nothing relies on defaults. Also added `iUnitCostPercent` in XML for exhaustiveness (value **100**) without changing its DLL default (**100**) since it was correct. Spotted thanks to the new Sevopedia Game Speed Chart. See: [KI#92](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#92---fixed-base-advciv-bug-of-iextrafreeoutsideunits-being-set-to-100-at-lower-than-marathon-game-speeds-spotted-thanks-to-the-new-sevopedia-game-speed-chart).
- **Timeline & pace**: fast-paced but long enough to play through. Start at **-50 000 BC**; on **Normal**, early turns advance by **5 000 years/turn**, then the step size **gradually decreases**. After a few dozen turns at **Normal** speed, you’ll be around the **Bronze Age**. Late-game turns are kept relatively brisk; the game ends at **2105 AD** (**turn 500** on Normal). See full game speed details in the **new Sevopedia Speeds category**.

## World Size/Scaling

- **New World Sizes**: `SAS24`, `SAS32`, `SAS40`, `SAS48`, **bigger than Huge** (Recommended for about 24, 32, 40, 48 players). They are [modified versions of the XXL World mod](/README.md#about-the-xxl-world-mod)'s largest maps. Recommended DLL to play these is the [48 Civs DLL](/_1_AdvCiv-SAS/Docs/README_Main_Changes_Guide.md#48-civs-dll) that is provided in AdvCiv-SAS. Also a new `Arena` World Size **smaller than Tiny**. See full World Sizes details in the **new Sevopedia W. Sizes category**.
- **Research — sea level**: `iResearchPercent = 100` for High / Medium / Low (**was** `117 / 100 / 87`). Sea level no longer alters research pace.
- **Research — world size**: `iResearchPercent` rescaled from `95 / 98 / 102 / 115 / 137 / 150` to `85 / 90 / 95 / 100 / 105 / 110` (so **Standard = 100**). Rationale: on larger maps the AI doesn’t start with proportionally more early cities, so heavier early research multipliers left it behind. We now tune tech costs directly in XML. **Examples** (illustrative, see Sevopedia/XML for current values): on **Standard**, `TECH_MATHEMATICS` = 250 (was 285) and `TECH_ALIEN_LIFE` = 16000 (was 18400).

- (Requires AdvCiv-SAS 5970+) **Expanded handicap difficulty ladder:** added **Rookie** below Settler and **Deity+** above Deity. Rookie gives new/casual players a softer entry point (AdvCiv-SAS Settler is harder and more gradual to other higher handicaps and less of a tutorial sandbox than base AdvCiv), while Deity+ gives expert players an extra challenge above Deity (AdvCiv-SAS Deity has less cheats and more gradual to previous handicaps).
- (Requires AdvCiv-SAS 5970+) **Cleaner handicap difficulty rating scale:** reworked `iDifficulty` from the old Settler-to-Deity `10, 20, 30, 40, 50, 60, 70, 80, 90` curve to a new Rookie-to-Deity+ `18, 26, 34, 42, 50, 58, 66, 74, 82, 90, 98` curve. This keeps Noble at `50` as the even/baseline difficulty, and places the new lower/higher difficulties around that scale without using `0` or values above `100` (`iDifficulty` is not the main source of direct AI bonuses or player penalties; those still come from fields like `iAIResearchPercent`, `iAITrainPercent`, maintenance, happiness, barbarian modifiers, and so on. Instead, `iDifficulty` is a general difficulty rating used by scoring, comparisons, and code that needs to average or estimate handicap strength).
- (Requires AdvCiv-SAS 5970+) **More robust handicap averaging:** updated DLL code that converts an averaged difficulty score back into a handicap. Previously, some code effectively averaged raw handicap enum/order IDs, which **only works reliably when difficulty rows are fixed and evenly spaced**. It now averages XML `iDifficulty` values and maps back to the nearest handicap, so added or future non-linear handicap levels behave according to the XML difficulty rating rather than accidental enum position.
- **No free techs** for AI or human.
- **Human costs fixed at 100 on all handicaps** for tech/unit (`iResearchPercent`, `iTrainPercent`, `iGPThresholdPercent` (great person) stay constant for the human). AI equivalents scale with handicap.
- `iBaseGrowthThresholdPercent` (base food required to grow a city): **Fixed at 100 on all handicaps.** Rationale: In base AdvCiv this value varies by difficulty to add a **human growth/expansion penalty** (and it can also shift early expansion pacing via settler cost logic tied to the initial growth threshold). Consequence: Removing this scaling makes **high handicaps** (e.g. Deity) **easier for the human player** (faster early growth/expansion than base AdvCiv), and makes **low handicaps** (e.g. Chieftain) **harder for the human player** (less of an early growth/expansion boost).
- **Unit maintenance** (`iUnitCostPercent`, `iAIUnitCostPercent`) aligned across handicaps for fairness; difficulty tuned through other params.
- **Unit supply parity:** Raised `iAIUnitSupplyPercent` from 50 to 100, so AIs no longer get half-price unit supply outside their borders; human and AI players now pay the same supply costs (unlike base AdvCiv).
- **Unit upgrade costs — aligned and reduced.** Increased `iAIUnitUpgradePercent` from **50 → 100**, so AIs now use the **same unit upgrade multiplier** as humans (no more half-cost AI upgrades). In parallel, `CvUnit::upgradePrice` now applies a global **modifier** (current value: **75%**, tunable), because the old human rate felt too expensive in play. Net effect: in base AdvCiv the AI paid **50%** and the human **100%** of the base upgrade cost; in AdvCiv-SAS, **both now pay 75%** of that cost.
- **No delayed starts**: `INCREASE_START_TURN` disabled; all handicaps start on **turn 0**.
- **AI worker speed**: `iAIWorkRateModifier` **0** on all handicaps.
- **AI starting workers**: `iAIStartingWorkerUnits` **0** on all handicaps.
- **Goodies (huts)**: remove **worker/settler** pops; rebalance barbarian outcomes (fewer strong, more weak at high handicaps); increase **experience** rewards at higher handicaps.
- (Requires AdvCiv-SAS 5970+) **Explicit and tighter barbarian handicap fields:** added **explicit values** for previously missing barbarian-related handicap fields (`iSeaBarbarianBonus`, `iBarbarianCityAttackBonus`, `iSeaBarbarianExtraMoves`) and tightened their low-difficulty curves. This avoids hidden XML/default behavior and reduces broad low-difficulty freebies across most of the ladder; for example, barbarian city attack modifiers no longer implicitly stay at `-25` through Noble, sea barbarian combat modifiers no longer implicitly stay at `-10`, and sea barbarian extra movement penalties no longer implicitly apply through Deity. The new curves concentrate strong protection on Rookie/Settler/Chieftain, then reach neutral values earlier.
- (Requires AdvCiv-SAS 5970+) **Lower and more gradual barbarian/animal handicap protection:** retuned several **barbarian** and **animal** handicap modifiers so only the lowest difficulties protect players as of now. For example, `iAnimalBonus`, `iAIAnimalBonus`, `iBarbarianBonus`, and `iAIBarbarianBonus` now follow a clean `-30`/`-20`/`-10`/`0` curve for Rookie/Settler/Chieftain/Warlord and then stay neutral at higher handicaps. In base AdvCiv, several of these protections lasted much farther up the ladder: `iAnimalBonus` still gave `-25` at Noble and remained non-neutral through Immortal, while `iAIAnimalBonus` and `iAIBarbarianBonus` stayed at `-25` all the way from Settler through Deity. This keeps the lowest difficulties forgiving, gradually less so, while removing broad barbarian/animal freebies from most difficulties.
- (Requires AdvCiv-SAS 5970+) **More difficulty-sensitive barbarian city creation:** reworked `iBarbarianCityCreationProb` from base AdvCiv’s very flat curve into a wider Rookie-to-Deity+ scaling curve. Base AdvCiv only rises from `4` on Settler to `8` on Deity, while AdvCiv-SAS now scales from `0` on Rookie to `40` on Deity+.
- **Handicap normalization (starting units)**: Reduced/standardized several handicap fields so they’re **the same at all handicaps**: `iStartingDefenseUnits`, `iStartingWorkerUnits`, `iStartingExploreUnits`, `iAIStartingDefenseUnits`, `iAIStartingWorkerUnits`, `iAIStartingExploreUnits`. This reflects recent AI improvements (workers, building choices, scrapping rules, unit mix) and aims to keep games fair while the AI remains competitive.
- **Difficulty scaling — `iAIHandicapIncrementTurns`**: **disabled (set to 0 on all handicaps)**. This removes the **per-turn** AI discount that made games progressively harder; with AdvCiv-SAS’s improved hammer efficiency, that scaling led to runaway AIs at high levels. Now AI cost/research **modifiers** come **only** from the static **handicap XML fields** (e.g., `iAITrainPercent`, `iAIBuildingCostPercent`, `iAIResearchPercent`, …), which have been **retuned to be linear by handicap (not per-turn)**—hard but fair. Practical effects: fewer excess units and lower bankruptcy risk, healthier mid→late-game economies, and challenge that relies on AI competency gains rather than exponential scaling.

## Terrains / Features

- Overall: **snow**, **desert**, and **water** tiles strengthened mainly via bonus/building changes (see Bonus/Buildings sections).
- **Floodplains persist after raze**: destroying a city does not remove Flood Plains, so previously-developed land can still stay profitable.
- **Ground explore/recon units**: can move through **all land terrains** from game start (including **peaks**) via `bCanMoveImpassable` (no water tiles). They also **ignore terrain movement costs** (not just the Renaissance Explorer). See **Sevopedia Terrain: Peak** or XML for details.
- **Other units** (example: Quechua Warrior): some specific units also have `bCanMoveImpassable` for **all land terrains**; see **Sevopedia Terrain Peak/Hill** entries or XML/docs for per-unit details.
- **Air explore/recon and land-air units**: the **Airship** (see its Sevopedia entry or this guide's military section) and **Gunship** can move through **all terrains, including peaks and water tiles** using `bCanMoveImpassable` and `bCanMoveAllTerrain`.
- **Movement restrictions (outside cultural borders)**: some units (e.g. chariots, some siege) **cannot enter forest/jungle** feature tiles outside borders. This restriction still applies even if the tile is roaded (base Civ4/AdvCiv logic reused). See Sevopedia Feature entries and the feature-category rework ([example-3](/_1_AdvCiv-SAS/Docs/README_Sevopedia_Reworks.md#example-3-features-category)) or unit pages for the exact list.
- **Forest/jungle combat penalties**: some units (notably UNITCOMBAT_MOUNTED_MELEE) now get forest/jungle penalties; UNITCOMBAT_MOUNTED_RANGED penalties are milder/limited (defense-only). See Sevopedia Feature entries ([example-3](/_1_AdvCiv-SAS/Docs/README_Sevopedia_Reworks.md#example-3-features-category)) and unit pages for the exact modifiers.
- **Terrain/feature modifiers matter more in combat**: many more **units** have **terrain/feature modifiers** (and sometimes plot-type like **Hill**). **Civ-specific units** generally have the **strongest modifiers**; **generic units** only get them when especially **relevant**. The goal is usually a "dominant terrain, minor terrains" approach (strong enough to matter, but not so strong that it replaces core roles like city-defense Longbowmen). See unit pages, Sevopedia Terrain/Feature pages ([example-9](/_1_AdvCiv-SAS/Docs/README_Sevopedia_Reworks.md#example-9-terrains-category), [example-3](/_1_AdvCiv-SAS/Docs/README_Sevopedia_Reworks.md#example-3-features-category)), and [other maps used for terrain modifiers](/README.md#other-maps-i-used-for-terrain-modifiers-for-civ-specific-units). **Update:** Now optional: players can now **selectively disable** terrain/feature attack modifiers or terrain/feature defense modifiers (e.g. Roman Legionaries stronger (+25%) on plains; chariots weaker (-50%) on jungle, etc.) via SAS defines. Implemented in `CvUnitInfo::getTerrainAttackModifier` / `CvUnitInfo::getTerrainDefenseModifier` (and feature equivalents).
- **Build: Remove Jungle** now yields **+20 hammers** on removal (see `BUILD_REMOVE_JUNGLE` in [CIV4BuildInfos.xml](/Assets/XML/Units/CIV4BuildInfos.xml)).
- **Feature: Jungle rework**: remove the food penalty (no food malus), increase unhealth to `iHealthPercent` **-50**, set feature defense `iDefense` **-25** (still +25% rival defense in features, so net 0 in jungle if you consider that baseline).
- **Feature: Fallout** `iHealthPercent` **-50 -> -75**.
- **Feature: Flood Plains** `iHealthPercent` **-40 -> -50**.
- **Terrain: Tundra** base yield adds **+1 commerce**.

## Bonuses

- **Grapes**: now improved by **Plantation** (not Winery). Happiness moved to **Grocer** (Grapes and Sugar both give happiness via Grocer). See Sevopedia/XML.
- **Hit bonuses** functionally disabled (kept late in tree, obsolete on reveal) to avoid LSystem issues; dependent assets rebalanced or left as‑is where appropriate.
- **Camel** replaces Hit Movies; spawns on **desert**. High food profile; supports camel‑mounted units (see Military section).
- **Bonuses — distribution (repartition) & yields (with stronger competition):** Adjusted **distribution/spawn chances** and **yields** across many bonuses.  
  **Yields:** formerly weak bonuses (e.g., **Silver**, **Deer**) are **stronger**; a few outliers (e.g., **Gold**) are **slightly weaker**; overall, bonuses grant **less commerce** to keep pacing gradual.  
  **Repartition:** **land food bonuses spawn less often** to create **real competition** for high-value spots; after testing, rates were raised enough that maps still feel lively. Several bonuses were **reworked** for clearer roles (e.g., **Sugar** gives **base hammer**; **Horse** gives **base food** on the tile but **hammer** once improved). **Deer** now **spawns more** (primary tundra/snow food) with **strong yields**, making **tundra cities** near Deer more viable. See **Sevopedia → Bonus** for current values and **[commit b841968](https://github.com/wonderingabout/AdvCiv-SAS/commit/b841968682f022e2cefbe1722efb3520492659b8)**; screenshots: **[5238](https://drive.google.com/file/d/1178did3ROEoIV_wbonpH91jZcdaNtzRg/view?usp=sharing)**, **[5241](https://drive.google.com/file/d/15acCk947FpTMmDJdzg6sMXCXOjYwiUB0/view?usp=sharing)**.
- **Water bonuses — increased but conservative spawns:** **Molluscs** and **Fish** now spawn **more frequently** (they’re "general" water resources). Rates were tuned with tests to avoid over-spawning—**significant yet conservative** increases. See **Sevopedia → Bonus** for current values.
- **Whales — buff & placement:** **Whales** have been **buffed** and can now **spawn on coast** as well as ocean. Details in **Sevopedia → Bonus → Whales**.
- **Horses — no tundra placement:** **Horses no longer spawn on tundra.** Food needs there are primarily handled by **Deer**. Concentrating Horses on non-tundra improves typical access patterns on most starts.
- **Edge-weighted bonus placement:** Shifted bonus distribution to **favor map edges** more (it was too **center-weighted**), reducing the advantage of central starts and making **edge starts more competitive**.
- **Desert rework — Camel replaces Movies:** On **desert**, **BONUS_CAMEL** now **replaces BONUS_MOVIES**, providing a **desert-friendly food bonus** and serving as a **mounted unit resource** that **competes with Horses** (see **[Military: camel changes](/README.md#military-tree-and-changes)**). Camel XML derives from Horse with AdvCiv-SAS adjustments; desert tiles become more viable via **high food yield**.
- **Horse frequency reduced:** With **Camels** added as an alternative mount, **Horse** **spawn frequency is reduced** to keep mounted-resource balance.
- **High-food animals slightly rarer:** Other **high-food land animals** are **slightly reduced** in frequency so **Camels** can be present without making food abundances trivial—keeping **competition for top sites** intact.
- Adjust some **animal Food Bonuses**' terrains/features for balance, accuracy, or thematic fit: **Sheep**: Grassland + Plains -> **Plains only** (Plains are weaker, Grassland food is overpowered and over-represented, real world population is mostly on savanna-like biomes and culturally closer to African, Western Asian, and Australian regions); **Pig** Jungle -> **Forest** (real world population is mostly around typical forest biomes, this helps counter-balance the Sheep removal from Grassland, and Jungle is stronger in AdvCiv-SAS (no inherent food-penalty) so it can accommodate this change).
- **Gemstones** can spawn on **flood plains**. See Sevopedia/XML.

## Traits

- Several Traits reworked. Weaker ones are buffed/modified; some previously OP ones are adjusted. Example: **Protective** is stronger; **Industrious** shifts away from "wonders faster" toward **worker/production-oriented** effects (elements formerly in Expansive). See in-game Sevopedia → Traits for current effects (+x culture / +x hammer / +x commerce as listed there).
- **Removed Trait — Expansive:** In AdvCiv-SAS, **Protective** is broadened from "just city defense" to an empire-wide protection theme: it now also covers health and no anarchy (political stability), and with **Imperialist** already handling expansion (settler discount + now also trade-route–scaled yields), **Expansive** no longer had a distinct role, was overlapping with the former, and was weak/volatile to balance → removed.

- **Trait icons (UI) & cleanup:** Updated the Sevopedia "char" (glyph/logo) for **Imperialist** and **Organized** Traits; commented out the obsolete **Expansive** Trait entries now that the Trait is removed. Implemented in [/Assets/Python/BUG/TraitUtil.py](/Assets/Python/BUG/TraitUtil.py).

## Civics

- **Civics rebalance (selective):** `CIVIC_REPRESENTATION` adjusted (unlocks earlier in AdvCiv-SAS; effects tweaked to fit timing; **Upkeep: Medium → High** to counter-balance its per-specialist scaling and thematically reflect the heavier apparatus needed to sustain such systems); `CIVIC_SLAVERY` buffed (**Upkeep: Medium → Low**); `CIVIC_CASTE_SYSTEM` nerfed (**Upkeep: Medium → High**) to curb early snowball; `CIVIC_VASSALAGE` buffed (**Upkeep: High → Medium**); and other changes. See Sevopedia/XML.
- **Civic reworks:** `CIVIC_WAGE_LABOR` (former Emancipation) retuned into an earlier **production/economy** civic for greater impact and historical accuracy; `CIVIC_SERFDOM` strengthened into a more **agrarian/food-focused** civic. Some religion civics are buffed/reworked to be refocused on religious play and to stay relevant longer; notably, `CIVIC_THEOCRACY` now uses `iCivicPercentAnger` = 200 (vs 400 on former civic_emancipation) (value info added to Sevopedia and Policy Advisor); economy civics are also more specialized/distinct, e.g. the old Environmentalism civic is replaced with `CIVIC_TRADE_BLOC` (earlier regional union/league-style trade civic); `CIVIC_PROTECTIONISM` is now about having **less trade**, not **no trade** (not isolationism); and `CIVIC_STATE_PROPERTY` now redistributes wealth via yield shifts (e.g. −20% hammers in the capital, +10% in all cities → effectively -20 + 10 = −10% in the capital, vs +10% in all other cities); plus other reworks such as for `CIVIC_NATIONHOOD`. Net effect: more viable paths, fewer must-picks.
- **Civic unlocks moved in the tech tree:** Some civics are available earlier/later than in base AdvCiv (e.g., **Slavery** moved to **Agriculture**). This improves historical pacing and balance/strategy variety (notably, moving Slavery out of the military line reduces the "everything funnels through Bronze Working" effect). See Tech Tree/Sevopedia/XML for current placements.
- **Civics order aligned to tech pace:** Reordered civics in XML so the **Policy Advisor** displays them in rough unlock order (UI only; no rules change). Examples: **Universal Suffrage** before **Police State**; **Caste System** before **Vassalage**; **Pacifism** before **Theocracy**.

## Technologies

- **Large reorder** for historical flow (e.g., Pottery before Wheel; Metal Casting before Currency and Bronze Working; Medicine earlier). Overall tech count stays similar to base (some low-value techs removed, with the **Future era expanded**). **Quotes/announcer lines** were reassigned to better fit the new placements (including Future era) and for accuracy. See [Assets rebalancing: Tech quotes](/_1_AdvCiv-SAS/Docs/README_Assets_Rebalancing.md#tech-quote-swaps-rework).
- Gameplay timings shift: **culture**, **slavery**, **plantations**, etc., often **earlier**.
- **Selective tech-trade bans for snowball techs:** Some high-impact techs are now flagged **"Cannot be traded"** (`bTrade = 0`) to curb extreme tech-brokering/"tech whoreism" chains (e.g. turning one breakthrough into a long series of trades), which AIs are especially vulnerable to and for balance. The goal is to tone down decisive, snowbally trades while still allowing normal tech trading (unless you enable the **No Tech Trading** game option, which disables it entirely). Info added in UI: see [UI (In-game)](/_1_AdvCiv-SAS/Docs/README_Main_Changes_Guide.md#ui-in-game) and [UI (Sevopedia)](/_1_AdvCiv-SAS/Docs/README_Main_Changes_Guide.md#ui-sevopedia) sections.
- **TECH_COST_NOTRADE_MODIFIER** **-23 → -20** (when No Tech Trading option is on). See [GlobalDefines_advc.xml](/Assets/XML/GlobalDefines_advc.xml).
- **Exponential pacing**: fewer late-era techs/unit types (Future era is the exception and is expanded), partly reflecting that ancient eras cover far more "years" than modern ones.
- **FirstFreeUnitClass** assignments reshuffled to diversify GP races (one GP type per tech; often earlier). *Exception:* the Future era has its own GP assignments.
- **Delayed Requirements**: Some techs (e.g. `TECH_MOUNTED_COMBAT`) can optionally be researched fairly early (e.g. at Classical Era) or skipped until significantly later when they become mandatory (e.g. mid-Medieval Era for `TECH_CHIVALRY`). This gives more strategic flexibility and research costs management, as these techs would be cheaper to research later on with a stronger economy, while shorter-term concerns have more room to be addressed sooner if desired.
- **Hastened Requirements**: Conversely, some techs (e.g. `TECH_CORPORATION`) are available earlier than their column in tech tree would suggest (since their requirements are few), for versatility and diverse strategical choices (thematically, it is also sensical as an alternate reality possibility).

## Eras

- **Future era** reworked into a full era with techs/units/buildings and its own music; begins with **tech_depopulation** as a balancing/historical nod. See Sevopedia/XML, and [Copyright and Disclaimer](/README.md#copyright-and-disclaimer).
- **Eras — research scaling**: `iResearchPercent = 100` (**was** `100 / 90 / 80 / 70 / 60 / 50 / 40` for all eras); `iTechCostModifier = 0`. Tech costs now come directly from TechInfos (no hidden per-era multipliers). Tech costs have been adjusted to match this change.
- **Eras — anarchy scaling**: `iAnarchyPercent = 100` for all eras (**was** `50 / 50 / 40 / 40 / 34 / 34 / 34`). In practice this yields about **2 turns** of anarchy when you change civics (or similar). Intent: add strategic weight without tedium; **indirectly buffs** Traits with **no anarchy** (e.g., Spiritual).
- **Eras — growth & production scaling flattened**: set `iGrowthPercent` (food-to-grow), `iTrainPercent` (unit cost), `iConstructPercent` (building cost), and `iCreatePercent` (project cost) to **100 for all eras** (**was** `100 / 100 / 100 / 90 / 80 / 70 / 60`). Removes late-era discounts that sped up mid/late game; pacing is now **predictable and gradual**, with balance handled directly via per-item `iCost` instead.
- **Eras — worker builds & improvements scaling flattened**: set `iBuildPercent` and `iImprovementPercent` from **100/100/100/90/80/70/60 → 100** at **all eras**. The original reason for the era-based scaling isn’t known; this change is intentional to keep tile-development pacing **predictable and consistent** across eras and to tune balance directly through base costs/worker rates rather than hidden era multipliers.
- **Eras — culture scaling flattened**: set `iCulturePercent` from **100/100/80/70/60/50/50 → 100** at **all eras**. Removes late-era culture acceleration so culture growth stays **consistent** and driven by **commerce/buildings**, not hidden era multipliers. In **AdvCiv-SAS**, if we ever want modern (or ancient) culture to weigh differently, we prefer adjusting the **explicit culture values on buildings/units** rather than reintroducing era-wide modifiers.
- Add **new XML field** `<Button>` for eras, to cleanly display them in Sevopedia and ingame without having to hardcode the path. Change in [CIV4GameInfoSchema.xml](/Assets/XML/GameInfo/CIV4GameInfoSchema.xml), [CIV4EraInfos.xml](/Assets/XML/GameInfo/CIV4EraInfos.xml), [CvEraMovieScreen.py](/Assets/Python/Screens/CvEraMovieScreen.py). See [README: Button for eras](/README.md#button-for-eras).

## Culture

- **Great Artist rebalance:** culture bomb strength reduced (**700 → 500 per era**), but long-term culture output increased (**12 → 18 culture/turn**). This strengthens long-term culture play while toning down early spikes.

## Religions

- **Religions roster:** added **Paganism**; removed **Confucianism** (chosen partly because it has fewer adherents in our framing). Rationale: a lineup that feels simpler and (to us) more historically fitting. We **keep Judaism** as the in-game representative of early **monotheism**, and we continue to prefer a **small set (~7) of religions** for clarity and efficiency.
- **Missionaries:** significantly **cheaper**.
- **Monasteries:** simplified — now **espionage** buildings with a **low Great Person rate**; **no longer obsolete**; **no longer give science** (keeps science buildings cleaner and the theming clearer)..
- **Shrines:** now require **(1) the religion present in the city** and **(2) the religion’s tech** (previously shrine TechPrereq was `NONE`). This makes shrines show up properly in the **Religion** Sevopedia building list and in the **tech tree**, and prevents "conquer (or any way to acquire) holy city ⇒ build shrine immediately" without the religion/tech. See [KI#12](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#12---shrines-now-require-their-religion), [KI#13](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#13---shrines-now-appear-in-tech-tree), and [README_Sevopedia_Reworks.md#example-5-religion-category](/_1_AdvCiv-SAS/Docs/README_Sevopedia_Reworks.md#example-5-religion-category).

## Corporations

- **Executives:** significantly **cheaper**.
- **Executives** now **require the same Tech(s)** to be built as their corresponding corporation's does (e.g. `TECH_MARINE_TECHNOLOGY` and `TECH_CORPORATION` for the Sid Sushi's executive unit; was `NONE`.) to be built. This simplifies the Sevopedia Unit Era grouping implementation so they are correctly listed (e.g. "at Modern Era", not at "No Technology Prequisite"), and executive's effective availability (i.e. not until the later eras). This also balances **rival city acquisition** so a player **cannot produce executives if we do not have the corresponding Corporation** in the city (mirroring the shrine prerequirement change).
- **Great Person variety:** each corporation uses **one distinct Great Person type** (no overlaps).

## Event Triggers and Events

- The standard new game mode (**Simple Game**) now has **Events Disabled** by default. We find they can be annoying, distracting, or frustrating; they do not offer a strong gameplay value, and setting up a Custom Game every time to disable them is tedious. Players can still enable events via the **Custom Game** mode: make sure the "No Random Events" option is unticked, or alternatively revert this change. Change in [CIV4GameOptionInfos.xml](/Assets/XML/GameInfo/CIV4GameOptionInfos.xml) (`GAMEOPTION_NO_EVENTS`'s `bDefault`: `0` -> `1`).
- Expose `CvEventTriggerInfo` and `CvEventInfo` **python getters** (done so SevoPedia Event Triggers can show them), and fix 3 incorrect ones (see [KI#122](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#122---fixed-while-adding-sevopedia-eventtriggerinfo-missing-getters-found-and-fixed-3-dll-python-binding-bugs)).
- Moved Vote XML **read time** from `CvXMLLoadUtility::LoadOptionalGlobals` to `CvXMLLoadUtility::LoadPreMenuGlobals` (similarly to votes else SevoPedia Event Triggers are not loaded in main menu's launch time and are empty in Sevopedia).

## Barbarians

- **Barbarians — stronger early game & smarter aggression**: economically **stronger in the early game**, growing at a pace closer to human/AI civs and **founding cities more often**, so they stay **relevant longer** before gradually fading by midgame. They still aren’t a full player (**no diplomacy**; **scattered independent cities** that collectively count as the Barbarian civilization). This pairs with the AI’s improved **hammer efficiency and availability** (e.g. and notably, far fewer scrapping issues): by engaging and "burning off" surplus early units (e.g., **ancient macemen, archers, longbowmen**), barbarians help prevent runaway over-production and early bankruptcies while empires keep growing. For definitions and exact knobs, see the Sevopedia **Barbarian Civilization** entry or XML: [CIV4CivilizationInfos.xml](/Assets/XML/Civilizations/CIV4CivilizationInfos.xml), [CIV4BuildingInfos.xml](/Assets/XML/Buildings/CIV4BuildingInfos.xml), and [CIV4HandicapInfo.xml](/Assets/XML/GameInfo/CIV4HandicapInfo.xml).
- Barbarian civ-specific **buildings** can be captured (`iConquestProb`) with **half the chance** of their generic building equivalent (since their cities are overall easier and lower risk to capture, it balances it out). See Sevopedia/XML.
- Barbarian **workers** are civ-specific and **cannot be captured** (prevents worker farming).
- Barbarian **workboat** is civ-specific mainly for cost/balance.

## Civilizations

- New civs: **Kingdom of Benin**; **Ireland**. See also [World map with civs](/README.md#world-map-with-civs).
- **Starting techs — reworked**: `TECH_AGRICULTURE` & `TECH_THE_WHEEL` are **no longer** starting techs; `TECH_POTTERY` **now is**. Even distribution across civs (**~14 per tech**). A **"pair"** = a civ’s two starting techs (unordered), e.g., **Native Americans** now `HUNTING + MINING` (fits Dog Soldier / Bronze Working). **Current** max pair frequency is **4** (not a hard cap). Done with **heavy assistance from ChatGPT-5** plus manual tuning. See: [Assets Rebalancing table (starting-techs (civ list & rationale))](/_1_AdvCiv-SAS/Docs/README_Assets_Rebalancing.md#after-most-recent-as-of-now-rework), or the values and rankings only in the new **Sevopedia Techs' Starting Techs Charts**.

## Leaders

- New leaders: **Ewuare**; **Grace O'Malley**, **Michael Collins**.
- Add support for optional **leaderhead art per era** with the new `EraArtDefineTags` and `EraArtDefineTag` XML fields; e.g., Gandhi's appearance and button changing at Ancient, Classical, Medieval, etc. The feature **is technically functional**, but **currently no leaderhead era art has been added**. Note: This is **per-player** not human-centered; e.g. even if the human player is at Medieval Era, if the rival Gandhi is still at Classical Era then Gandhi will use the Classical Era (if set). If none is set or disabled in SAS defines, the art defaults to the base `ArtDefineTag`. Note 2: due to a Civ4 limitation, if a leader (e.g., Gandhi) is used by several players (e.g., in Custom Game), only the first of these leaders' current era is used to select the art for all instances of that leader regardless of their own progress (e.g., if Gandhi 1 is at Medieval and Gandhi 2 and 3 are still at Classical, then Gandhi 2 and 3 will use Gandhi 1's Medieval art). See [README.md (EraArtDefineTags and EraArtDefineTag for leaderhead era art)](/README.md#eraartdefinetags-and-eraartdefinetag-for-leaderhead-era-art) and [KI#120](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#120---documented-known-limitation-per-era-leader-art-shows-the-lowest-index-players-era-when-the-same-leader-is-assigned-to-multiple-players).
- Each leader now has **distinct leader music** (previous BTS/AdvCiv shared tracks split per leader). See [Copyright & disclaimer](/README.md#copyright-and-disclaimer). Note: switched **De Gaulle** from Louis' music in Base AdvCiv to Base Civ4's Napoleon's music, and added an AdvCiv-SAS specific new music for Napoleon so each leader has its own specific music, see [KI#98](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#98---fixed-base-advciv-issue-of-de-gaulle-using-louis-music-instead-of-the-music-common-to-de-gaulle-and-napoleon).
- **Trait overhaul:** Overhauled for historical fit + gameplay relevance. **Examples**: Julius Caesar Organized + Imperialist → Aggressive + Imperialist *(military conqueror; offensive pressure)*; Saladin Protective + Spiritual → Spiritual + Charismatic *(religious leader with battlefield charisma; warmonger in Civ IV terms)*; Hatshepsut — Spiritual + Creative → Spiritual + Financial *(historical emphasis on trade/wealth; modest buff in practice)*. Full before/after with rationale in  [Assets Rebalancing table (Leader Traits)](/_1_AdvCiv-SAS/Docs/README_Assets_Rebalancing.md#leaders-traits-rework), or the values and rankings only in the new **Sevopedia Traits' Charts**. *Note: leader "flavors" unchanged.*
- Barbarian **leader button/icon** is distinct (no longer Genghis Khan’s).

## Diplomacy

- `BBAI_DEFENSIVE_PACT_BEHAVIOR` **disabled (0)** to restore default BTS behavior (pacts break on DoW).
- **Voluntary** vassals are **permanent** (thematically a full culture absorption/merger).
- **Permanent Alliance** moved off the main military path (was on both **Communism** and **Fascism**) to reduce military beeline dominance and respect "one tech per major unlock" consistency.

## Buildings

- **Granary:** cheaper, slightly weaker food retention (partly to balance the Slavery/Hurry changes). Civ-specific granaries unchanged (so their relative advantage increases). Note: the Incan granary (Qullqa) was reworked toward a production/growth role rather than a culture role.
- **Water building line reworked/buffed** *(Goal: make **water tiles more viable** while requiring you to work them for the benefits; yields apply per worked tile, not passively from routes.)*:
  - **Harbor** (first water building): health/food oriented.
  - **Lighthouse:** gold oriented (**+x commerce** from worked tiles where applicable).
  - **Port (new):** production oriented (**+x hammer** from worked tiles where applicable).
  - **Moai Statues:** world wonder with broader water synergy; somewhat stronger overall.
  - **Customs House:** buffed.
- **Great Person Buildings** (e.g. Academy, Scotland Yard) now **require a tech** (e.g. `TECH_LITERATURE`, `TECH_MONARCHY`) so they appear in the tech tree and in the Sevopedia Building's corresponding group rather than in the "No Tech Prerequisite" group. This may make strategic considerations more specialized and help slightly curb military beelines in particular.
- Fewer **national** wonders and relatively more **world** wonders.
- **Palace-likes** (Forbidden Palace, Versailles, etc.) mostly reworked as **world** wonders with tech prereqs so they appear in the tech tree and AIs don’t build them too early. (Barbarian Palace unchanged; used as a balance lever.)
- Many world-wonder effects simplified/retuned; overall costs adjusted (often slightly lower).
- Most `BonusProductionModifiers` (e.g., +100% with Marble) are greatly reduced. Example: Parthenon is now +25% with Marble (was +100%). Wonder costs were also adjusted to fit this change.
- Removed buildings that didn’t carry their weight or distorted pacing (e.g., **Space Elevator**, **West Point**).
- **Iron Works:** now requires **both Coal and Iron empire-wide**; city requirement count lowered; cost up moderately (e.g., 700 -> 800).
- **Heroic Epic** (`iMilitaryProductionModifier`): **100% → 50%**. The always-on military production boost was too strong and could let AIs run away; this tones it down while keeping the building relevant.
- Some **civ-specific** buildings replaced (especially late-game or underwhelming ones). Example: Russian **Research Institute** replaced with **Gord** (castle-based, earlier impact). See [Assets rebalancing: Civ-specific buildings rework](/_1_AdvCiv-SAS/Docs/README_Assets_Rebalancing.md#civ-specific-buildings-rework) for detailed rationales and before/after changes table.
- Added `BuildingClassRequired` chains to curb spam and improve AI focus: e.g., **Drydock** now requires **Port**.
- **Culture rebalance:** Trimmed incidental culture from science buildings (e.g., **Library: +2 → +1**) and from some non-purely-cultural national wonders (**Heroic Epic**, **National Epic**, **Forbidden Palace**). In exchange, **buffed/reworked culture-focused buildings** (e.g., **Colosseum** and civ variants) to keep the culture path meaningful.

## Units (Common)

- New **optional XML field**: `ObsoleteTech` **for units**: if set, this unit **can no longer be produced once we have discovered this tech** (e.g. Ancient Maceman (`UNIT_WARRIOR`) obsolete at `TECH_BRONZE_WORKING`). This XML field is **optional**: if **not set** (`<ObsoleteTech>NONE</ObsoleteTech>`, e.g. for the `UNIT_SETTLER`) or missing entirely (better **inter-mod portability**), a **fallback default of `NONE`** is provided anyway, so the unit is never obsolete, meaning no tech discovery will stop allowing to produce it (base AdvCiv/Civ4 behaviour). This new XML field allows to **avoid unit clutter in later eras** (no more **aberrant obsolete units in later eras** (e.g. Ancient Maceman produced in Medieval Era; Pikemen in Industrial Era)) being **produced** or shown in the **Production Chooser Bar**. **AI** becomes **naturally stronger** as a result of these changes. Additionally, this also helps set **clean scrapping rules** to computationally cheaply detect and remove old units for the AI's military composition, or **further enhancements**. **Sevopedia** Blurbs with links fully implemented in [CvGameTextMgr.cpp](/CvGameCoreDLL/CvGameTextMgr.cpp). See [README: ObsoleteTech for units](/README.md#obsoletetech-for-units). See also [KI#93](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#93---enhanced-new-optional-xml-fields-eg-obsoletetech-for-units-to-fix-aberrant-unit-choice-or-scrapping).
- **Build timing** tweaks for some units for historical pacing; see the Technology Advisor.
- Detailed civ-specific unit documentation now lives in [Assets rebalancing: Civ-specific units rework](/_1_AdvCiv-SAS/Docs/README_Assets_Rebalancing.md#civ-specific-units-rework), covering civilian and military civ-specific units with before/after/rationale rows for the current rework pass. For updated in-game data, see also Sevopedia Unit.

## Civilian Units

- `DOW_UNIT_CAPTURE_CHANCE` **0 → 50**: workers can be captured **on the turn war is declared**, same as other turns. Aligns with the base capture chance and rewards opportunistic warfare.
- **Worker cost reduced** (see XML/Sevopedia).
- **Settler** is now a **national unit** (**max 1 per player**). This prevents AI from building two at once or building a new one while an existing settler is travelling, improves hammer/food efficiency, and raises the stakes on timing/escorts. See [KI#37](/_1_AdvCiv-SAS/Docs/README_Known_Issues.md#37---workaround-and-rebalanced-ai-building-2-settlers-very-inefficiently-at-same-time-or-when-a-settler-unit-already-exists---now-settler-is-a-national-unit-only-1-allowed-per-player).
- **Scouts**, **Explorers**, **Airships**, and **Robotic Recon Drones** are now **national units** (**max 2 each per player**): reduces AI scout overproduction, keeps unit maintenance more efficient, and makes early scouting choices a bit more strategic.
- **Explorer-class** units can **attack**, but with low strength.

## Military & related units

- **Camel-mounted line:** Added new camel units (Camel Archer, Camel Knight, Camel Cuirassier) tied to `BONUS_CAMEL` as an alternative to horse units, and reworked Arabia’s civ-specific unit as the Hajjan, a Camel Knight variant.
- **Unit Combat Types** expanded/refined (e.g., archers split into short/long/crossbow; mounted split into `UNITCOMBAT_MOUNTED_MELEE` vs `UNITCOMBAT_MOUNTED_RANGED`). Lets counters be more specific (e.g., pikes vs mounted-melee but not vs mounted-ranged). Some civ-specific units shift types accordingly (e.g., **Ballista Elephant** is mounted-ranged while **War Elephant** is mounted-melee). **Airship** becomes **Recon** (no longer sees subs; later recon can).
- **Melee Unit Combat Type split (rework + balance change):** `UNITCOMBAT_MELEE` is now split into `UNITCOMBAT_MELEE_POLEARM` (melee units that attack from some distance with a lance-like weapon; e.g. Spearmen, Pikemen) and `UNITCOMBAT_MELEE_SHOCK` (close-contact/brawl; e.g. Axemen, Swordsmen, Macemen). For example, after this change, axes still counter polearms but are only slightly stronger than swords; pikemen are no longer worse than axemen which was very odd considering cost and tech requirements difference. Civ-specific variants such as the Babylonian Bowman, or simply the generic Crossbow, now only apply vs. `UNITCOMBAT_MELEE_SHOCK` (i.e., weak vs pikes). **General idea for melee units:** Axeman > Spearman; Crossbow > Axeman/Swordsman/Maceman; Pikeman > Crossbow; Pikeman ≈ Axeman; Maceman > Pikeman. Civ variants retagged to match gear (e.g., Persia Immortal → polearm). **General idea for mounted units:** Spearman > Chariot (`UNITCOMBAT_MOUNTED_MELEE`); generic Horse/Camel Archer (`UNITCOMBAT_MOUNTED_RANGED`) > Spearman; generic Horse/Camel Knights (`UNITCOMBAT_MOUNTED_MELEE`) < Pikemen (`UNITCOMBAT_MELEE_POLEARM`); generic Horse/Camel Cuirassiers > Pikemen (`UNITCOMBAT_MELEE_POLEARM`). See Sevopedia/XML.
- **Reassignments:** e.g., **Mechanized Infantry** counts as **Armored** (not "Gun") for counter logic.
- **City defenses** affect all units, including gunpowder and later. See `bIgnoreBuildingDefense` in `CIV4UnitInfos.xml`.
- **Collateral** values normalized (fixed cases like **Stealth Bomber** having higher base collateral than its limit). Most units now show a **collateral damage limit** (starting at 0 for many) (hidden for civilian units since they don't use it). **Collateral promotions 1/2** are broadly available where eligible. Info shown in Sevopedia (Unit chart/Special Abilities) and unit hover panels.
- **Promotions retuned:**
  - Some buffed (e.g., **Woodsman**), some reworked/nerfed (e.g., **Hills Master 2 and 3**), some prereqs changed (e.g., **Archers** can take **City Bombard**; **Logistics** (formerly Commando) requires **Combat 1** or **City Raider 1**; higher-tier promotions (**City Raider/Garrison/First Strike 3**) also need **Combat 2**).
  - **Logistics** now at **The Wheel** (was Military Science).
  - Removed redundant promotions (e.g., **Tactics**), to avoid broken retreat odds.
- **Upgrade paths** simplified and made more sensible:
  - Mounted tiers upgrade along mounted lines (e.g., **Chariot → Horse Archer**; **War Elephant → Knight**).
  - Cross-mount upgrades allowed (**Horse <-> Camel** where logical).
  - **Longbows/Crossbows/Macemen/Pikemen → Musketmen** (more historical/balanced; reduces abuse by **Cuirassiers**, which are slightly stronger but more expensive).
  - **Dragoons → Tanks** now.
- **Naval upgrades** cleaned up; **Ironclad** can enter ocean and is buffed, serving as a real upgrade from Frigates. See [README: military tree](/README.md#military-tree-and-changes) for full upgrade maps.
- **Support costs:** some late/robotic units have **no support cost**; **Privateer** now **has support** (was free).
- **Weak/late civ-specific** units **replaced by earlier/impactful ones** (e.g., Dutch East Indiaman → Dutch Swift Worker; German Panzer → Hussar; Mongol Keshik → Khishigten — better window with the Ger, longer medieval span, higher hammer rate, and more historical; English **Redcoat** → **Yeoman archer** — crossbow-based, fits England’s defensive profile, and gives cover when short on early bonuses; Holy Roman Landsknecht → Houfnice — cannon-based aligns with HRE’s aggressive profile, hits the tight medieval window, and avoids Landsknecht’s obsolescence/balance issues (anti-melee niche fades; weak vs. Cuirassiers)). Detailed civ-specific unit documentation now lives in [Assets rebalancing: Civ-specific units rework](/_1_AdvCiv-SAS/Docs/README_Assets_Rebalancing.md#civ-specific-units-rework). For updated in-game data, see also Sevopedia Unit.
- **Broad unit reworks/rebalances:** e.g., French **Musketeer** — Move 2 → 1; gains anti-mounted and small collateral kit for a stronger Renaissance+ role (especially vs Cuirassier units and their upgrades).
- **Bonus requirements relaxed (selective):** Some units now accept additional **bonus** prerequisites where appropriate (to avoid odd lockouts). Examples: **Samurai** can use **Copper** as well as Iron; **Swordsman-class civ-specific variants** (e.g., **Roman Legionary**, **Incan Quechua**) can also use **Copper** (only the **Gallic Warrior** could before; which seemed inconsistent and unbalanced without a clear reason why: now equalized); **Crossbowman-class civ-specific variants** (e.g., **Zhuge Nu**, **Disciplined Crossbowman**) Bronze/copper hardware was common (Benin bronzes; Chinese bronze/iron mix long before ubiquitous steel) and to buff them; Holy Roman **Houfnice** can also use **Copper** (historically plausible bronze artillery). The **generic Swordsman** and the **generic Cannon** remain **Iron-only** to preserve strategic pressure. Thematically consistent. See Sevopedia/XML.
- **Naval/animal** units broadly rebalanced; see Sevopedia/XML for details.
- Units more **versatile:** e.g., Swords defend better; Archers can attack more credibly.
- **Barracks** now require **Hunting**.
- **Warrior / "Ancient Maceman"** rebalanced: higher base strength, lower city-combat, gains **vs Animals** bonus; cost slightly up.
- **Archer / Longbowman:** both available earlier with lower base strength and cost but with new stat mixes; **Archer** no longer upgrades to late units; **Longbow** upgrades to **Crossbow** (earlier availability).
- **Spearman line:** base Strength **4 → 5**; anti-mounted role strengthened; civ-specific variants further differentiated (see Sevopedia/XML).
- **Grenadiers & Bazooka (formerly Anti-Tank Infantry)**: rebalanced into **hybrid, versatile "foot-bomber" roles**.
- **Gun units buffed (relative to pre-gun)**: units that **carry firearms** (e.g., **Musketman**, **Horse/Camel Cuirassiers**) are **relatively stronger** than their pre-gun counterparts.
- **Air** unit costs generally increased.
- **Stealth** removed where it didn’t make sense (kept for subs, spies, stealth bombers).
- **Caravel** removed (privileged border passing felt wrong); shift that niche to **Privateer** (buffed) and streamline the era’s naval roster.
- **Nuclear Submarine** removed (under-impactful relative to complexity).
