- ai aggregates: they aim to represent the behaviour of a leader regardring some ai attributes that characterize said behaviour. For example conqueror, tech_broker, diplomat etc. An aggregate is the accumulation/mathematical merging of the impact of several ai attributes to reflect the currently selected leader and how said leader fares as compared to all leaders in a median-based sorting (we explain later in the text how we compute these). For example a leader like montezuma will have say 90% conqueror while gandhi will have say 0% of conqueror (may not be exact, is just to illustrate). Here is an example of how an ai aggregate is represented and implemented in our code:

To go back on the representation of aggregates in our (me and becomingthrough chatgpt anyways) code, it is done at module initilization before the class code of SevopediaLeader. In our code, the leader data initialization and preprocessing logic is done efficiently and only once at module load before the class code. We first define some static data such as the examples above, and for example also other reused strings such as header names, etc. Then, in the cache_ai_value_ranges function, we load and cache the ai attributes data of all leaders in AI_VALUE_RANGES and AI_SORTED_VALUES. Caching before class load is done to increase performance, especially but not only on less performant devices, perhaps, ideally. 

Then, our pre-processing of the leaders data involves the get_percentile function that returns the percentile value of the ai attribute that is passed as parameter, between 0 and 100 (normalization) in a median or average based distribution (currently only median is used). For example warrand 50 would be for example the 3% percentile among all leaders, while the warrand 400 would be close to example the 95% percentile among all leaders for the raw ai attribute value of warrand.

This get_percentile function is used both for the processing of raw ai attributes and of the ai aggregates, which take then different paths (even though we display both for each leader and quite similarly eventually, it is done quite differently too):

1) for raw ai attributes we rely on the function cache_ai_attribute_data, to simply cache each leader's raw (ex: for warrand 400 for Gandhi, 50 for Montezuma) and percentile-normalized values (for example 95%, 3%) for every attribute.for all leaders.

2) for ai aggregates (ex: conqueror, diplomat, etc.), we also use ai attributes data, but we do not call the aforementionned cache_ai_attribute_data. Instead we use the cache_ai_aggregate_scores. For each ai attribute that composes the ai aggregate (ex: conqueror is composed of getConquestVictoryWeight, getMaxWarRand, getMaxWarNearbyPowerRatio (may or not be accurate, it is to illustrate and as an example)), we first look at if the ai attribute is directly correlated (more warrand (higher value) equals more conqueror aggregate behaviour), or "inversely" correlated with this aggregate (more warrand (higher value) equals less conqueror aggregate behaviour). For example Gandhi which is known to be peaceful has a warrand of 400, while Montezuma which is knows to be a warmonger has a value of 50. By cross-checking and confirming a similar distribution (low value for warmongers, high(er) value for peaceful personalities), we can infer warrand is invertly correlated with the conqueror behaviour, so we invert it, and set the invertFlag to true. This means that unlike for what we did before for raw ai attributes that we do not invert, here in ai aggregates processing, we instead invert the ai attributes, so for example warrand 400 will now be instead 100 - 95 = 5% percentile among all leaders, while warrand 50 will be for example 100 - 3 = 97% percentile among all leaders. This ensures that the warrand attribute is accurately correlated with the conqueror aggregate, as now a higher percentile of warrand will result in a higher conqueror aggregate score as epxlained later in this paragraph. It is important and key to note that ai attributes are aggregate specific, for example a buildunitprob attribute (example, may be accurate or not, is to illustrate) that would be directly correlated with more units being produced (higher value equals more units produced), would as a consequence be directly correlated to some war aggregates (conqueror or/and other aggregates) as more units equal more war, but less in a builder (example) aggregate for example, as more units being built equals less buildings and wonders, so we would invert (true) this buildunitprob aattribute for the builder aggregate, while we we would not (false) invert it for some war aggregates (conqueror or/and other aggregates). We repeat this process for all the other ai attributes that compose the ai aggregate, and for each aggregate too. Attributes in ai aggregates are also characterized by a weightFloat (positive value between 0 and 1), that aim at representing the partial contribution of said attribute to the aggregate, for example in the (example may be accurate or not, is to illustrate) conqueror aggregate, getConquestVictoryWeight attribute may be an important part of conquering behaviour so it has a maxium weight of 1.0, while getMaxWarNearbyPowerRatio is important to an extent to the conquerir behaviour but less, so it would have a quite high weight but lower than 1.0, for example 0.6.
