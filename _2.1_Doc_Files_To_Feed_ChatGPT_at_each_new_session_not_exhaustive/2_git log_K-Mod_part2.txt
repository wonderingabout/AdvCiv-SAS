commit d1f55f29aa0daeec6157ee1e2b473ef37fe7ebe0
Author: karadoc <karadoc@gmail.com>
Date:   Fri Sep 30 00:54:07 2016 +1000

    Adjusted AI_counterPropose to prefer an unfair deal over no deal.
    
    When considering the iValueForThem > iValueForUs case, we were trying to
    get as close to a balanced trade as possible. However, this meant that
    when the fairest deal would favour a human 'them', the AI would decline
    to offer any deal at all.
    
    I've altered the behaviour so that the top priority is to favour 'us',
    and getting a fair deal is secondary; rather than the other way around.
    This means fair AI-AI deals might be less likely; but counter-proposals
    to human trade offers will work better. And that's the main aim.

commit 68bf5845c92505fdca826f6d8869a399fbead281
Author: karadoc <karadoc@gmail.com>
Date:   Wed Sep 28 22:25:35 2016 +1000

    Rewrote CvPlayerAI::AI_counterPropose to get more balanced deals.
    
    AI_counterPropose now worked by evaluating all possible trade items
    before adding them to the trade, so that items can be added in the order
    most likely to result in a balanced deal.
    
    In addition to this, the AI will now be slightly more generous to humans
    in the sense that the automatic counter proposals from human players
    will no longer add gold if the AI would accept the deal without gold.
    These automatic counter proposals are the only time the AI will allow a
    discounted trade with a human. This change means that the AI may offer a
    trade deal which favours a human player even if the human player has
    enough gold to balance the deal. Previously, if the human player had a
    lot of gold on hand, the AI would never offer a discounted trade. (Also
    note that in the original BtS code, the AI would never offer a
    favourable deal to a human under any circumstances.)
    
    This was a significant rewrite, so there might be some bugs. However, my
    main concern is that the new version might actually work 'too well' in
    the sense that AI players might make significantly more trade deals with
    each other, and thus be stronger than before. We'll see if further
    changes are needed.

    Tweaked the handling of happiness and growth in AI_yieldValue.
    
    There is a complex interplay between the approximate food-per-turn given
    by iAdjustedFoodPerTurn and the approximate future happiness given by
    iFutureHappy. The growth value adjustments from iFutureHappy were sometimes resulting
    in some dubious plot choices (because high food plots can end up giving
    lower growth value than low food plots).
    
    Estimated future happiness now depends on a slower estimate of
    food-per-turn, so that the 'future happiness' is more likely to be an
    underestimate rather than an overestimate.
    
    I've also increased the devalue rate. (i.e. decreased the value of
    having very high food-per-turn.

commit 5b28f89dc3d5fd4c86faa91e1d505664c92404f8
Author: karadoc <karadoc@gmail.com>
Date:   Tue Sep 16 19:44:19 2014 +1000

    Restructured pathfinder usage to not use getPathLastNode.
    
    FAStarNode pointers returned by the pathfinder become invalid when the
    pathfinder is destroyed. Because of this, I consider it bad practice to
    use these pointers. In fact, the the pathfinder shouldn't allow any kind
    of external access to them at all. Not only could misuse result in
    segfaults, but the pointers could also be misused in a way to mess up
    the internal state of the pathfinder...
    
    So, in this patch, almost all usage of FAStarNode has been removed from
    outside of the pathfinder. This will have no effect on the execution of
    the game, but it may help to reduce bugs in the future.
    
    I've disabled CvUnit::getPathLastNode and
    CvSelectionGroup::getPathLastNode, and replaced their usage with new
    pathfinder methods such as KmodPathFinder::GetFinalMoves and
    KmodPathFinder::GetPathTurns.
    
    There are only a couple of places where FAStarNode is still used. One is
    in CvUnitAI::AI_solveBlockageProblem, which doesn't use the
    KmodPathFinder, so I probably won't change it. The other is
    CvUnitAI::AI_considerPathDOW. To correct this, I intend to create a new
    method to give access to the completed path without exposing the
    internal nodes. (But I haven't done that yet.)
    
    I've also removed the KMOD_PATH_FINDER define and preprocessor
    conditions. The original bts pathfinding code has been deleted.

    The open list in KmodPathFinder is now a vector, to improve speed.

commit 89c902c322270c5b3a7789dc34a8bebc91855e8e
Author: karadoc <karadoc@gmail.com>
Date:   Mon Dec 17 18:16:37 2012 +1100

    Fixed a BtS pathfinding bug which caused inaccurate path turns.
    
    Previously, if units with different speeds were grouped together moved with different movement costs, the pathfinder would fail to calculate the correct number of path turns. The most obvious example is when moving a group with mixed speeds along a railroad.
    
    The bug was caused by the fact that the path nodes only store the lowest
    moves value for the group, and so if the unit with more moves also has a
    greater move cost, then the pathfinder will effectively be using the
    lower moves with the higher movement cost.
    
    I've fixed this by recalcuating the exact number of move for each unit
    for the final turn of the path whenever there are mixed movement costs
    involved. I expect this to have negligible speed inpact.
    
    --
    
    Also, I've changed CvSelectionGroup::maxMoves to be the minimum of the
    group's max moves rather than the maximum. This seems far more sensible
    to me.

    Implemented an all new version of CvPlayerAI::AI_techBuildingValue.

    scaled based on the production cost of the building, and a few other
    factors.

    will be removed at some later time.

    buildings enabled by techs. It was completely separate from
    CvCity::AI_buildingValue, and it ignored _most_ building functionality.
    I've decided it would be better to use the thorough calculations already
    present in AI_buildingValue. This should make the AI a bit smarter, and
    the code more easy to update. But it might run slightly slower.
    
    Note: There are some cases where this new kind of evaluation is
    significantly flawed. For example, Assembly Line enables factories and
    coal power plants; since these two buildings are highly dependant on one
    another, their individual evaluation will greatly undervalue them.
    Another example: catherals can't be built in every city, but this
    function will evaluate them as if they could, thus overvaluing them. But
    still, the original code was far worse - so I think I'll just tolerate
    such flaws for now.

commit 89d942d0765767820b3820daab66fd53b9754db9
Author: karadoc <karadoc@gmail.com>
Date:   Mon Sep 24 15:23:52 2012 +1000

    Redesigned the evaluation of attack_city units.
    
    The previous system was focus around a somewhat artibrary global target
    bombard rate. The value of bombard units was boosted significantly while
    below the target. Unfortunately, if siege units were powerful enough to
    be chosen even when above the bombard target, then the AI would end up
    with too many siege units - and also, the calculation of total bombard
    was only a rough approximation, distrorted by the presence of defensive
    units with bombard.
    
    The new system does not use a bombard target, and instead focuses on the
    number of units with a combat-limit of less than 100. The value of the
    unit is reduced if there are already too many combat-limited units. The
    reduction in value primarily targets the bombard component and the
    collateral damage component.
    
    The new system is still a bit of an ugly kludge, but I think it will
    have fewer systematic weaknesses than the old system.

commit 575791c4b1c2a55aa2092133f2790f6213d1fbad
Author: karadoc <karadoc@gmail.com>
Date:   Thu Jun 21 10:21:54 2012 +1000

    Completely rewrote the attitude cache system.
    
    Cache is now recalculated on the spot when something happens to
    invalidate it.
    
    CvPlayerAI::AI_invalidateAttitudeCache has been removed and repalced
    with CvPlayerAI::AI_updateAttitudeCache.
    
    The original system (from the 'CAR' mod) was hideously prone to OOS
    bugs. I'd fixed many of the bugs in the past, but there were still more,
    and it was extremely difficult to write robust code using that system.
    (Here a couple of lingering bug examples from the old system.
    CvTeam::declareWar didn't clear cache for the friends of the war target.
    CvTeam::signDefensivePact didn't clear cache for the rivals of each
    team. The crux of the problem is that in the CAR system, it was the
    programmers responsibility to know exactly how and when attitude could
    be affected in any way, and then clear the cache when it was affected.
    If the programmer overlooked anything, it could cause OOS.)
    
    The new system is more rigid in when the cache is recalculated; and so
    similar 'bugs' will now no longer result in OOS. At worst they will
    result in out-of-date attitude ratings. In fact, this is such a minor
    effect that I've deliberately removed the cache update from a few places
    where I think it is unimportant.
